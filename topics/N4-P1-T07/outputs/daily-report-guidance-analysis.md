## 二、问题本质分析

### 2.1 这些问题属于什么类型？

**核心判断**：

| 问题类型 | 是否属于 Project 生命周期 | 说明 |
|----------|--------------------------|------|
| **日常汇报类** | ❌ **不属于** | 不涉及 Project 的创建/执行/监控/关闭/归档 |
| **职责查询类** | ❌ **不属于** | 不涉及具体 Task 执行，只是查询 Agent 职责范围 |
| **状态同步类** | ❌ **不属于** | 不涉及 Project 状态变更，只是同步信息 |

---

### 2.2 这些问题应该在哪里规划引导？

**结论**：应该在 **Agent 个人工作汇报机制** 中规划，不在 Project 生命周期介入入口中。

**原因**：

| 原因 | 说明 |
|------|------|
| **不涉及 Project 操作** | 这些问题不改变 Project 状态，只是查询/汇报 |
| **属于 Agent 日常沟通** | Harold 与 Agent 的日常同步，不是项目管理介入 |
| **已有现成机制** | Heartbeat 晨会报告、MISSION_BOARD、memory 文件已有 |
| **需要独立引导类型** | 应作为"日常汇报引导"，与 Project 生命周期引导分离 |

## 三、日常汇报引导设计

### 3.1 引导类型定义

**引导类型 7：日常汇报引导** ⭐（新增）

**触发条件**：Harold 询问 Agent 工作状态/职责范围

**触发关键词**：
- "你现在负责什么？"
- "你今天的工作是什么？"
- "你昨天的工作是什么？"
- "早会"
- "现在干嘛呢？"
- "最近在忙什么？"

---

### 3.2 引导内容设计

**引导选项模板**：

```
我理解您想了解我的工作状态。请选择查询类型：

A. 查看我的职责范围（IDENTITY.md + 当前负责的Project）
B. 查看今日任务清单（MISSION_BOARD §三 进行中区）
C. 查看最近工作记录（memory/YYYY-MM-DD.md）
D. 生成晨会/夕会报告（Heartbeat 报告机制）

回复 A/B/C/D 或描述具体需求。
```

**选项数量**：4 个（A/B/C/D）

---

### 3.3 各选项对应动作

| 选项 | 查询内容 | 数据源 | 输出格式 |
|------|----------|----------|----------|
| **A** | Agent 职责范围 | IDENTITY.md + PROJECT-CHARTER | 职责表格 |
| **B** | 今日任务清单 | MISSION_BOARD.md §三 进行中区 | 任务列表 |
| **C** | 最近工作记录 | memory/YYYY-MM-DD.md | 工作日志 |
| **D** | 晨会/夕会报告 | Heartbeat 状态 + MISSION_BOARD | 结构化报告 |

## 五、dispatch 如何识别日常汇报类问题？

### 5.1 关键词匹配规则

```python
def is_daily_report_intent(user_message):
    """判断是否为日常汇报类意图"""
    
    # 关键词列表
    daily_report_keywords = [
        "负责什么", "职责", "负责",
        "今天的工作", "今日工作", "今天的任务",
        "昨天的工作", "昨日工作", "昨天的任务",
        "早会", "夕会", "晨会",
        "现在干嘛", "在忙什么", "最近在忙",
        "工作状态", "任务状态"
    ]
    
    # 匹配规则
    for keyword in daily_report_keywords:
        if keyword in user_message:
            return True
    
    return False
```

---

### 5.2 dispatch 路由逻辑

```
用户消息到达
    ↓
意图识别
    ↓
if 日常汇报关键词匹配:
    路由到 ⑦ 日常汇报引导
elif Project 生命周期关键词匹配:
    路由到 Project 生命周期引导（①-⑥）
else:
    路由到 chat 或其他意图
```

## 六、与 Heartbeat 汇报机制的关系

### 6.1 日常汇报引导 vs Heartbeat 汇报

| 维度 | 日常汇报引导 | Heartbeat 汇报 |
|------|-------------|---------------|
| **触发方式** | Harold 主动询问 | 自动触发（早 7:00/晚 9:00） |
| **触发时机** | 随时 | 固定时间 |
| **内容类型** | 查询型（职责/任务清单/历史记录） | 报告型（晨会/夕会完整报告） |
| **引导必要性** | ⚠️ 可能需要引导 | ❌ 不引导（自动生成） |

---
