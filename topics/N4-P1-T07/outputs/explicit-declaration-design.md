## 一、Harold 核心修正 ⭐

### 1.1 原设计问题

**原设计**：
- **明确归属 → 不引导**：用户说"关闭 P015" → 直接执行，不输出任何确认信息 ❌
- **紧急情况 → 不引导**：用户说"立刻处理" → 直接执行，不输出任何确认信息 ❌

**Harold 指出的问题**：
- 没有显式声明 → 用户不知道Agent的选择是什么
- 可能出现理解偏差 → 用户说错了也无法纠正
- 缺少完整信息输出 → 用户看不到完整的项目名/Task ID

---

### 1.2 Harold 修正后的设计原则

**核心原则**：**对于任何明确的、紧急的情况，都需要显式声明你的选择，让用户知道你的选择是什么。这就是"显式声明"。**

**修正后的设计**：
- **明确归属 → 不引导，但需要显式声明执行动作** ⭐
- **紧急情况 → 不引导，但需要显式声明执行动作** ⭐
- **所有执行动作 → 都需要显式声明完整信息** ⭐

---

**示例 1：关闭项目**

| 阶段 | 原设计 | Harold 修正后 |

## 二、显式声明设计规范

### 2.1 显式声明格式

**格式模板**：

```
正在执行：{完整对象名称}（{补充信息}）

示例：
正在关闭：ZT-P015 NUCLEUS 4.0 项目（Harold审批流程）
正在执行：N4-P1-T07-T02 Dispatch引导入口设计（状态：执行中）
正在查询：ZT-P015/N4-P1-T07 的Task列表（共 3 个Task）
```

---

### 2.2 显式声明触发场景

| 场景 | 是否显式声明 | 显式声明内容 | 时机 |
|------|-------------|-------------|------|
| **① 用户明确归属（关闭 P015）** | ✅ 必须显式声明 | 完整项目名 + 执行动作 | 执行前 |
| **② 用户明确Task ID（继续 T07-T02）** | ✅ 必须显式声明 | 完整Task名 + Task ID + 执行动作 | 执行前 |
| **③ 用户紧急处理（立刻处理）** | ✅ 必须显式声明 | 紧急标记（P0） + 执行动作 + 确认信息 | 执行前 |
| **④ Agent理解意图后（引导选择后）** | ✅ 必须显式声明 | 完整对象名 + 执行动作 | 执行前 |
| **⑤ Agent上报阻塞** | ✅ 必须显式声明 | 阻塞Task ID + 阻塞原因 + 处理选项 | 上报时 |

---

### 2.3 显式声明与引导的关系

**核心原则**：

```
任何执行动作 → 必须显式声明 ⭐

引导 → 是为了确认意图
显式声明 → 是为了确认执行动作

两者独立，但都需要：
- 引导 + 显式声明 = 用户意图不明确
- 显式声明（无引导） = 用户意图明确
```
