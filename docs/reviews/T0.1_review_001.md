### 3.2 为什么这么做

**选择 exec tool 的理由**：

| 方式 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| **exec tool** | 无需改造、与现有 Skills 一致 | 启动开销（~50ms） | ✅ 选择 |
| import 内联 | 无启动开销 | 需要 OpenClaw 改造 | ❌ |
| 文件触发 | 解耦彻底 | 轮询开销、延迟高 | ❌ |

**核心决策依据**：
- OpenClaw 现有工具链无需改造
- 与现有 model-usage、skill-creator Skills 保持一致
- 符合 Harness Engineering 原则（工具链集成）

### 3.3 关键技术点

1. **{baseDir} 变量替换**
   - OpenClaw 自动替换为 Skill 基准目录
   - 示例：`{baseDir}/scripts/xxx.py` → `/opt/homebrew/lib/node_modules/openclaw/skills/xxx/scripts/xxx.py`

2. **错误传递机制**
   - stdout：正常输出（JSON/Text）
   - stderr：错误信息
   - 返回码：区分错误类型（1=参数错误，2=文件错误，3=处理错误，4=系统错误）

3. **Token 效率**
   - Python 脚本不占用 context window
   - LLM 只需调用脚本，无需生成代码

### 3.4 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| Python 脚本启动开销 | 性能影响 | 简单操作直接在 Skill 描述，复杂操作才用脚本 |
| 错误处理不完整 | 系统不稳定 | 定义标准错误传递规范 |
| 与现有 Skills 不兼容 | 功能受损 | 新增 scripts/ 目录，不修改现有 frameworks/ |
