## 二、竞品方案分析

### 2.1 claude-analysis (Python + SQLite)

**项目地址**：https://github.com/jnspo1/claude-analysis

**架构特点**：
- Python 实现
- SQLite 缓存数据库
- FastAPI dashboard
- 完整的工具调用分析

**优势**：
- ✅ 增量更新（file_cache 表记录已处理文件）
- ✅ 预计算 aggregates，查询性能好
- ✅ 完整的可视化 dashboard

**劣势**：
- ❌ 违反"零数据库"原则（使用 SQLite）
- ❌ 复杂度高（需要维护 DB schema）

### 2.2 ccwasted (Rust + 纯静态分析)

**项目地址**：https://github.com/fyso-dev/ccwasted

**架构特点**：
- Rust 实现
- 纯静态分析（No AI）
- CLI 工具
- Claude Code StatusLine 集成

**优势**：
- ✅ 无数据库，符合"零数据库"原则
- ✅ 性能优异（Rust 内存安全 + 零拷贝）
- ✅ CLI 工具，集成简单

**劣势**：
- ❌ 每次都要重新扫描所有文件（性能瓶颈）
- ❌ 功能相对单一（主要关注 token 浪费）

### 2.3 jsonl-tools-mcp (MCP Server + Configurable)

**项目地址**：https://github.com/antonlvovych/jsonl-tools-mcp

**架构特点**：
- MCP (Model Context Protocol) 服务器
- 可配置的日志分析
- 自动 schema 检测
- 语义字段映射

**优势**：
- ✅ 灵活的配置和查询能力
- ✅ 自动 schema 检测，适应性强
- ✅ 语义字段映射，使用友好

**劣势**：
- ❌ 需要运行额外的 MCP 服务器
- ❌ 复杂度高，可能过度设计

## 五、sessions/*.jsonl 字段规范

### 5.1 OpenClaw sessions/*.jsonl 结构

基于实际样本分析，OpenClaw sessions/*.jsonl 包含以下事件类型：

#### message 事件
```json
{
  "type": "message",
  "id": "26d261f7", 
  "parentId": "5130cbc7",
  "timestamp": "2026-04-05T05:15:30.098Z",
  "message": {
    "role": "assistant",
    "content": [...],
    "api": "openai-completions",
    "provider": "bailian-coding",
    "model": "glm-5",
    "usage": {
      "input": 0,
      "output": 0, 
      "cacheRead": 0,
      "cacheWrite": 0,
      "totalTokens": 0,
      "cost": {"total": 0}
    },
    "stopReason": "stop"
  }
}
```

#### session 事件
```json
{
  "type": "session",
  "id": "343d02f5-c428-4eaa-ae12-2dceeb776a4d",
  "cwd": "/Users/haroldtsui/Workspaces/openclaw/main/60_Agents/cqo",
  "timestamp": "2026-04-05T05:15:29.876Z"
}
```

### 5.2 Monitor 需消费的字段映射

| 监控指标 | 数据源字段 | 必需性 | 降级策略 |
|---------|-----------|--------|----------|
| **user_message_intervals** | `message.role="user"` + `timestamp` | 必需 | 跳过缺失 timestamp |
| **model_usage** | `message.role="assistant"` + `provider/model/usage.totalTokens/cost/stopReason` | 必需 | usage 缺失时设为 0 |
| **error_rate** | `message.stopReason != "stop"` | 必需 | stopReason 缺失时视为正常 |
| **session_count** | `type="session"` | 必需 | 跳过无效 session 事件 |
| **thinking_level** | `message.thinkingLevel` | 可选 | 缺失时不统计 |

**关键说明**：
- **user_message_intervals**：这是**用户交互间隔**，不是系统 heartbeat 周期
- **系统 heartbeat 监控**：由 `CycleScheduler` 负责（参见 T2）

### 5.3 与 OpenClaw 实际格式的适配差异

**假定存在的字段**（需要通过真实样本校正）：
- `message.usage.totalTokens`：某些模型可能不返回
- `message.cost.total`：成本计算依赖模型定价
- `message.thinkingLevel`：仅在启用 thinking 时存在

**降级策略**：
- 字段缺失时跳过该事件（不崩溃）
- malformed JSON 行直接跳过
- 统计 malformed 行数用于可观测性

## 六、增量消费模型结论（P0 问题修复）

### 6.1 增量更新机制要求

**硬性要求**：增量消费必须基于以下机制之一，**不能只依赖 mtime**：

- **选项 A**：`filepath + offset`（字节偏移）
  - 记录每个文件的最后处理位置
  - 每次从 offset 之后开始解析
  
- **选项 B**：`event_id`（稳定事件标识）
  - 为每条事件生成唯一 ID（如 `session_id + line_number`）
  - 维护已处理 event_id 集合避免重复

**原因**：OpenClaw sessions/*.jsonl 是 **append-only** 文件，mtime 变化时文件会增长，重跑全文件会导致历史事件重复统计。

### 6.2 状态存储路径对齐（P0 问题修复）

**路径规范**：
- **聚合日志**：`logs/YYYY-MM-DD.jsonl`（系统根目录）
- **最新状态**：`logs/latest.json`（系统根目录）
- **watermark 状态**：`cycles/scheduler_state.yaml`（monitor_watermarks 字段）

**禁止行为**：
- ❌ 创建独立的 `scanner_state.json`
- ❌ 在技能目录下创建 `logs/` 子目录

**理由**：避免日志树分裂，确保与 ARCH/WBS v1.1 约定一致。
