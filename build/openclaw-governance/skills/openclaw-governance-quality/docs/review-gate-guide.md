## 一、概述

### 1.1 定义

**Review-gate** 是强制审查检查机制，用于增强 NUCLEUS L0-L3 Review 体系，防止未审查的任务发布。

### 1.2 核心原则

```
┌─────────────────────────────────────────────────────────────────┐
│                    Review-gate 核心原则                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. 强制检查：所有任务必须通过 Review-gate 才能关闭             │
│   2. 自我认证防护：Builder ≠ Reviewer                           │
│   3. 审查完整性：必须有审查产物                                  │
│   4. 零问题警告：全通过的审查标记为可疑                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 与现有 Review 体系的对应

| 现有 L0-L3 Review | Review-gate 增强 |
|-------------------|------------------|
| L0: Harold 全量 Review | + 强制检查产物完整性 |
| L1: Harold + Agent | + 自我认证防护 |
| L2: Agent 方案 + Harold 确认 | + DOD 验证检查 |
| L3: Agent 执行 + 后置 Review | + 零问题警告 |

## 二、检查项

### 2.1 四项检查

| 检查项 | 说明 | 错误码 |
|--------|------|--------|
| **审查产物完整性** | 检查是否有 review.md 文件 | 2 |
| **自我审查检测** | 检查 Builder ≠ Reviewer | 3 |
| **DOD 验证** | 检查 DOD 是否已验证 | 4 |
| **零问题审查** | 检查可疑的"橡皮图章"审查 | 5（警告） |

### 2.2 检查流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    Review-gate 检查流程                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   [1/4] 审查产物完整性                                          │
│       ↓                                                         │
│   查找 *review*.md 文件                                         │
│       ↓                                                         │
│   [2/4] 自我审查检测                                            │
│       ↓                                                         │
│   从 TASK-CARD 提取 Task PIC                                    │
│   对比审查产物中的 Reviewer                                     │
│       ↓                                                         │
│   [3/4] DOD 验证                                                │
│       ↓                                                         │
│   检查 TASK-CARD 中的 DOD 部分                                  │
│   验证是否有 ✅ 标记                                            │
│       ↓                                                         │
│   [4/4] 零问题审查                                              │
│       ↓                                                         │
│   检查审查产物的详细程度                                        │
│   标记可疑的"通过无问题"审查                                    │
│       ↓                                                         │
│   [汇总]                                                        │
│       ↓                                                         │
│   通过 / 失败                                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 三、使用方法

### 3.1 基本用法

```bash
# 验证单个任务
./validate-reviews.sh /path/to/task-dir

# 详细输出
./validate-reviews.sh /path/to/task-dir -v

# 帮助信息
./validate-reviews.sh --help
```

### 3.2 返回码

| 返回码 | 含义 | 处理建议 |
|--------|------|----------|
| 0 | 验证通过 | 可以关闭任务 |
| 1 | 任务目录不存在 | 检查路径 |
| 2 | 缺少审查产物 | 添加 review.md |
| 3 | 检测到自我审查 | 更换审查者 |
| 4 | DOD 验证缺失 | 完成 DOD 验证 |
| 5 | 零问题审查警告 | 人工确认审查质量 |
| 6 | 配置文件错误 | 检查 TASK-CARD.md |

### 3.3 集成示例

```bash
# 在任务关闭前执行
if validate-reviews.sh "$TASK_DIR"; then
    echo "可以关闭任务"
    # 执行关闭逻辑
else
    echo "Review-gate 失败，请检查"
    exit 1
fi
```

## 四、审查产物规范

### 4.1 文件位置

```
task-dir/
├── TASK-CARD.md
├── review.md              # 推荐：主审查文件
├── review-code.md         # 可选：代码审查
├── review-standard.md     # 可选：标准审查
└── deliverables/
    └── review-*.md        # 可选：多审查文件
```

### 4.2 内容要求

**必须包含**：
- Reviewer 名称
- 审查日期
- 审查结论（PASS/FAIL）
- DOD 验证结果

**推荐包含**：
- 发现的问题
- 改进建议
- 审查详情

### 4.3 示例

```markdown
# Review Report

> **Reviewer**: 银月 (main)
> **Review Date**: 2026-03-23
> **Status**: ✅ PASS

## 审查结论

任务已完成审查，所有必须标准已满足。

## DOD 验证

| 标准 | 状态 | 备注 |
|------|------|------|
| DOD-1 | ✅ | 已验证 |
| DOD-2 | ✅ | 已验证 |

## 问题列表

1. [Minor] 建议添加更多注释
2. [Suggestion] 可考虑优化性能

*审查人: 银月 | 审查时间: 2026-03-23*
```

## 五、与自动化级别的适配

### 5.1 Review-gate 行为

| 自动化级别 | Review-gate 行为 |
|------------|------------------|
| **L0-L2** | 必须等待 Harold 审批 |
| **L3** | 后置 Review，异常上报 |
| **L4** | 自动通过，记录日志 |
| **L5** | 无需 Review-gate |

### 5.2 自动化集成

```bash
# 根据 automation_level 决定行为
case "$AUTOMATION_LEVEL" in
    L0|L1|L2)
        # 强制等待 Harold 审批
        validate-reviews.sh "$TASK_DIR" --require-human
        ;;
    L3)
        # 后置 Review
        validate-reviews.sh "$TASK_DIR" || report_anomaly
        ;;
    L4|L5)
        # 自动通过
        log_review_gate_skipped "$TASK_DIR"
        ;;
esac
```

## 六、常见问题

### Q1: 审查产物应该放在哪里？

**答**：推荐放在任务根目录，命名为 `review.md`。也可以放在 `deliverables/` 下。

### Q2: 一个任务可以有多个审查吗？

**答**：可以。脚本会检查所有 `*review*.md` 文件。

### Q3: 如何避免自我审查检测？

**答**：确保 TASK-CARD 中的 Task PIC 与审查产物中的 Reviewer 不同。

### Q4: 零问题审查是什么？

**答**：通过了审查但没有任何问题或建议，可能是"橡皮图章"审查，脚本会警告。
