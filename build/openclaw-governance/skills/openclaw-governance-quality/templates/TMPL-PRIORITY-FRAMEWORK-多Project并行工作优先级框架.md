## 零、术语定义

> 所有文件中出现的以下术语，以本定义为准。

| 术语 | 定义 |
|---|---|
| **Project** | 具有独立目标、资源和时间线的工作单元，由 Project PIC 负责，对应 PROJECT-CHARTER.md |
| **Topic** | Project 下的专题工作单元，由 Topic PIC 负责，对应 TOPIC-BRIEF.md |
| **Task** | Topic 下的具体执行项，由 Task PIC 负责，记录于 MISSION_BOARD.md，对应 TASK-CARD.md |
| **Deliverable** | Task 的具体交付物，部分需要 Harold Review，在 TASK-CARD.md 中声明 |
| **PIC** | Person / Agent In Charge，对应层级的唯一责任人 |
| **Harold Review** | Harold 对 Deliverable 的审核行为，属于验收动作 |
| **Harold Decision** | Harold 对需要人类拍板的决策事项的裁定行为 |
| **PMO** | 项目管理办公室职能，由银月承担，负责协调和仲裁，不担任任何 Project PIC |

## 一、优先级分级体系

### 🔴 P0 · 紧急关键 (Critical & Urgent)
- **定义**：直接影响业务生存、系统稳定性或重大合规风险的任务
- **响应时限**：立即处理（< 2小时）
- **示例**：系统崩溃修复、安全漏洞处理、法律合规紧急事项、核心业务流程阻断

### 🟡 P1 · 重要高优 (Important & High Priority)
- **定义**：影响核心业务目标达成或重要里程碑的关键任务
- **响应时限**：24 小时内启动
- **示例**：关键功能开发、重要客户交付、战略项目推进、资源瓶颈解决

### 🟢 P2 · 常规重要 (Important but Normal)
- **定义**：对业务有积极影响但不紧急的改进和优化任务
- **响应时限**：本周内安排
- **示例**：功能优化、文档完善、流程改进、技术债务清理

### ⚪ P3 · 低优先级 (Low Priority)
- **定义**：锦上添花的改进或探索性任务
- **响应时限**：资源空闲时处理
- **示例**：非核心功能增强、实验性技术探索、体验微调

## 二、P 值指定权与复议权

> 解决「谁有权定 P 值」的问题。

| 优先级 | 指定权 | 复议权 | 最终裁定 |
|---|---|---|---|
| **P0** | Harold 或银月确认方可生效 | 任何 Agent 可提出异议 | Harold |
| **P1** | Task PIC 自定，银月备案 | 银月可复议 | 银月 |
| **P2 / P3** | Task PIC 自定 | 无需复议 | Task PIC |

**特殊规则**：
- Harold 直接指令自动获得 P0，无需走指定流程
- 安全相关事件自动升级为 P0，由银月确认后生效

## 三、状态标记与 P 值的联动规则

> 状态标记 `[!][?][V][x]` 与 P 值是两个独立维度，需联合使用才能完整描述任务状态。

| 状态标记 | 含义 | P 值行为 | 银月动作 |
|---|---|---|---|
| `[!]` | Harold 待决（Review 或 Decision） | **继承**原 Task P 级，不自动升级 | 确认已记录在 MISSION_BOARD 七区，静默等待 |
| `[?]` | 阻塞，需要资源介入 | **至少升至 P1**，阻塞会影响整体进度 | 立即介入，按 P 值高低依次调度 |
| `[V]` | 待验收 | **继承**原 Task P 级，不改变优先级 | 组织验收，通过后改为 `[x]` |
| `[x]` | 完成待归档 | **降至最低**，知识沉淀可在空闲时处理 | 确认知识沉淀完成后归档 |

**多任务同优先级处理规则**：
- 相同 P 级的多个 `[?]` → 按截止日期最近者优先
- 相同 P 级的多个 `[V]` → 按 Task 进入队列时间最早者优先

## 四、多 Project 冲突解决原则

### 1. 时间维度优先级
- **截止日期驱动**：距离截止日期最近的 P0/P1 任务优先
- **里程碑依赖**：影响其他项目关键路径的任务优先
- **资源窗口期**：利用特定时间窗口（如市场机会、人员可用性）的任务优先

### 2. 业务价值优先级
- **收入影响**：直接产生收入或避免重大损失的任务优先
- **战略对齐**：与年度/季度战略目标高度一致的任务优先
- **客户影响**：影响核心客户满意度或留存的任务优先

### 3. 资源约束优先级
- **人力稀缺性**：需要特定稀缺技能的任务优先安排
- **成本效益**：ROI 最高的任务优先
- **并行可行性**：可以与其他任务并行执行的任务优先安排

## 五、Agent 间协作优先级规则

### 1. 三层 PIC 责任制
- **Project PIC**：对 Project 整体目标和交付负责，银月**不担任**任何 Project PIC
- **Topic PIC**：对 Topic 内所有 Task 的推进负责
- **Task PIC**：对单个 Task 的执行和 Deliverable 质量负责
- **银月（PMO）**：负责跨 Agent 任务协调、优先级仲裁和进度监控，**不介入具体执行**

### 2. 资源竞争处理
- **P0 任务抢占**：P0 任务可以中断任何 P1/P2/P3 任务
- **同级任务协商**：同优先级任务由相关 Agent 协商，银月仲裁
- **Harold 时间分配**：按 P 值和紧急程度排序，由银月统一管理 Harold 的 Review/Decision 队列

### 3. 信息同步机制
- **状态实时更新**：所有 Agent 必须实时更新 MISSION_BOARD 状态标记
- **阻塞立即上报**：任务阻塞必须立即标记 `[?]` 并通知银月
- **变更及时通知**：P 值变更必须通知所有相关方

## 六、优先级动态调整机制

### 1. 触发条件
- **外部环境变化**：市场、政策、技术环境重大变化
- **内部状态变化**：资源可用性、团队状态、技术风险变化
- **反馈循环**：Harold Review 意见、数据指标、评审结果

### 2. 调整流程
1. 相关 Agent 评估调整必要性
2. 向银月提交调整建议（在 MISSION_BOARD 七区记录）
3. 银月组织快速评估（< 4 小时）
4. 更新 MISSION_BOARD 并通知相关方

### 3. 调整频率限制
| 优先级 | 调整频率上限 |
|---|---|
| P0 | 随时可调整 |
| P1 | 每日最多 1 次 |
| P2 / P3 | 每周最多 1 次（周一规划时） |

## 七、优先级执行保障

### 1. 承诺机制
- **能力匹配**：Agent 只承诺在能力范围内可完成的任务
- **缓冲时间**：任务排期必须包含 20% 缓冲时间应对不确定性
- **拒绝权利**：Agent 有权拒绝超出能力范围的不合理优先级要求

### 2. 银月巡检机制

> 取代原「每日站会」，适配全 Agent 体系。

| 巡检项 | 频率 | 动作 |
|---|---|---|
| 全量 Heartbeat（读取所有 Agent MISSION_BOARD） | 每日一次 | 银月主动读取，汇总异常上报 Harold |
| Harold 待决超时检查 | 每日一次 | 超过「期望回复时间」未回复，银月主动提醒 Harold |
| P1 任务无进展告警 | 每日一次 | P1 任务连续 3 日无进展，银月介入评估 |
| 优先级漂移检查 | 每周一次（周一） | 评估是否有任务 P 值需要调整 |
| 知识沉淀完成率回顾 | 每周一次（周五） | `[x]` 任务中知识沉淀未完成的比例 |

### 3. 问责机制
- **透明记录**：所有 P 值决策和执行偏差必须记录在 MISSION_BOARD
- **根因分析**：重大优先级执行偏差需进行根因分析，结果沉淀至 Wiki
- **持续改进**：基于执行数据每季度优化本框架

## 八、特殊情况处理

### 1. Harold 直接指令
- **自动 P0**：Harold 的直接指令自动获得 P0 优先级，无需走指定流程
- **快速通道**：绕过常规评估流程，立即执行
- **事后补录责任**：**执行该指令的 Agent** 负责在 MISSION_BOARD 中补录 Task 记录；银月在下次 Heartbeat 巡检时核查补录是否完成

### 2. 紧急安全事件
- **自动 P0**：任何安全相关事件自动升级为 P0，由银月确认后生效
- **全员响应**：所有 Agent 暂停当前 P1/P2/P3 任务，优先处理安全事件
- **事后复盘**：事件处理完成后 24 小时内完成复盘，结果沉淀至 Wiki

### 3. 资源严重不足
- **最小可行集**：聚焦最核心的 P0 任务，暂停所有非必要任务
- **外部求助**：启动外部资源协调机制
- **范围调整**：与相关方协商调整任务范围或延期，由 Harold 最终裁定

## 九、Harold Review 机制

### 触发方式（B + C 结合）
- **Task 级别**：在 TASK-CARD.md 的 Deliverable List 中标注 `[HR]` 需要 Harold Review
- **Project 级别**：在 PROJECT-CHARTER.md 中定义重大里程碑的 Harold Review 节点

### Harold Review 队列管理
- 银月统一管理 Harold 的 Review 队列，按 P 值和提交时间排序
- 所有待 Harold Review 的事项集中记录在 MISSION_BOARD 七区
- Harold Review 完成后，Task PIC 负责更新 MISSION_BOARD 和 TASK-CARD

## 十、框架维护与演进

- **版本控制**：本框架采用语义化版本控制
- **定期评审**：每季度由银月组织框架有效性评审，结果提交 Harold 确认
- **持续优化**：基于实际使用反馈持续迭代，任何 Agent 均可提出改进建议
- **新 Agent Onboarding**：新 Agent 的 IDENTITY.md 必须包含对本框架的引用，并在首次激活时由银月确认已读取并理解
