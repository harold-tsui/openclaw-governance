## 一、Review 级别定义

| 级别 | 名称 | Harold 动作 | 适用阶段 |
|---|---|---|---|
| **L3** | 全量审阅 | Harold 整体 approve 通过，自行逐条 review；**沉默不视为通过，持续等待** | 建立期 · 新类型首次出现 |
| **L2** | 抽样审阅 | Harold 随机审阅 20-30% | 过渡期 · 同类已有先例 |
| **L1** | 异常审阅 | 银月发现异常时直接上报 Harold | 巡航期 · 已有充分先例 |
| **L0** | 免审 | Harold 不介入，银月验收即可 | 巡航期 · Harold 明确授权 |

### L3 通过标准补充
- Harold 以整体 approve 方式通过，Harold 自行逐条 review
- **沉默不视为通过，持续等待**
- **银月负责按 Heartbeat 周期定期提醒，直至 Harold 明确表态**

## 三、Harold Review 时限与决策累积规则

### 3.1 Rule 1：正常 Review 时限

| Review 级别 | 正常时限 | 说明 |
|---|---|---|
| **L3** | **7 天** | 全量审阅需充分时间评估 |
| **L2** | **3 天** | 抽样审阅工作量较小 |
| **DA 提议** | **5 天** | 授权降级提议需评估历史表现 |

**时限计算起点**：Deliverable 标记为 `[!]` 并进入 MISSION_BOARD §四 待响应评审区的时间。

### 3.2 Rule 2：L3 超期处理（激活机制→待决堆积）

**核心原则**：不开「最佳实践自动走」的口子，保证决策库完整性。

```
L3 Deliverable 提交（标记 [!]）
    ↓
Day 0-7：正常等待期
    ↓
Day 7+：触发「激活机制」
    ├─ 银月在 HEARTBEAT 中标记该事项为「🔥 超期待决」
    ├─ 每 24 小时提醒一次（而非 Heartbeat 周期）
    └─ 同时询问 Harold：「是否暂时降级至 L2 处理？」
        ├─ Harold 同意降级 → 按 L2 抽样处理，但保留 L3 标签用于统计
        └─ Harold 未回应 → 继续堆积，不自动推进
```

**待决堆积管理**：
- 超期事项在 MISSION_BOARD 中单独分区「§八 超期待决区」
- 堆积数量影响系统健康度评分（Heartbeat Step 5 评估）
- 季度复盘时分析堆积根因（Harold 时间不足？任务优先级混乱？）

### 3.3 Rule 3：P0 紧急任务特殊处理

| 场景 | 处理方式 |
|---|---|
| **事前已知 P0** | Review 级别默认为 L2，事后 48 小时内补 Review |
| **执行中升级为 P0** | 立即标记 `[!]` 并通知 Harold，但 Harold 可选择「事后 Review」 |
| **事后 Review 时限** | P0 完成后 48 小时内必须完成 Review，否则进入「§八 超期待决区」 |

**P0 事后 Review 重点**：
1. 紧急处理是否符合预期
2. 是否有可沉淀的应急决策（生成 DL 或 LL）
3. 是否应调整该类型的默认 Review 级别

## 四、Lesson Learn 机制

> **设计理念**：通过 Lesson Learn（LL）机制实现自我纠错，每个错误都是系统进化的机会。

### 4.1 LL 条目结构

```markdown
### LL-[编号] · [条目标题]

| 字段 | 内容 |
|---|---|
| **编号** | LL-[三位数字，如 LL-001] |
| **类型** | Agent误读DL / DA失效 / Task返回重做 / 决策矛盾 / 其他 |
| **来源任务** | [TASK_ID] |
| **发现时间** | [DATETIME] |
| **错误现象** | [发生了什么，客观描述] |
| **根本原因** | [为什么会发生，深度分析] |
| **关联DL** | [相关的 DL 编号，如有] |
| **纠正措施** | [立即采取的修复行动] |
| **教训要点** | [提炼的核心教训，一句话] |
| **适用范围** | [此教训适用于哪些场景] |
| **状态** | 🟡 分析中 / 🔄 纠正中 / ✅ 已归档 / 📎 已整合入DL |
| **整合DL编号** | [如已整合，指向新/修订的 DL] |
```

### 4.2 自动触发条件

以下任一情况发生时，**自动触发 LL 流程**（无需等待 Harold 指示）：

| 触发条件 | 触发时机 | LL 优先级 |
|---|---|---|
| **Agent 误读 DL** | Harold Review 指出 Agent 引用的 DL 不适用或理解错误 | P1（24h内起草） |
| **DA 失效** | 授权降级后的任务出现重大偏差，触发 DA 暂停 | P0（12h内起草） |
| **Task 返回重做** | Deliverable 被 Harold 打回要求重做（非小修订） | P1（24h内起草） |
| **决策矛盾** | 同一类场景 Harold 给出与 DL 记录相矛盾的决策 | P0（12h内起草） |
| **客户/外部投诉** | 交付物引发外部投诉或重大负面反馈 | P0（12h内起草） |

### 4.3 LL 与 DL 的联动流程

```
LL 触发
    ↓
Agent PIC 起草 LL 条目（使用 4.1 模板）
    ↓
LL 分析会（Agent PIC + Topic PIC + 银月，必要时邀请 Harold）
    ↓
制定 DL 更新策略
    ├─ 方案A：修订现有 DL（LL 揭示原 DL 表述不清或有漏洞）
    ├─ 方案B：新增 DL（LL 揭示新场景未被覆盖）
    ├─ 方案C：废止旧 DL（LL 揭示原 DL 已不适用）
    └─ 方案D：暂不更新 DL（纯执行层面问题，与决策无关）
    ↓
执行 DL 更新（如需）
    ↓
LL 状态更新为「📎 已整合入DL」或「✅ 已归档」
```

**LL 分析会输出物**：
- LL 条目完整填写
- DL 更新方案（含具体修订内容）
- 是否需要重做已交付内容
- 预防措施（如何避免同类错误）

## 五、授权降级发起条件（v2.0）

### 5.1 综合命中率计算方式

授权降级不再仅依赖「同类先例数量」，而是采用**综合命中率**评估：

```
综合命中率 = (DL命中率 × 0.4) + (质量分 × 0.3) + (时效分 × 0.3)

其中：
- DL命中率 = 正确引用DL次数 / 总执行次数
- 质量分 = (无偏差任务数 + 轻微偏差任务数×0.5) / 总任务数
- 时效分 = 按时完成任务数 / 总任务数
```

### 5.2 降级申请条件

| 条件 | 要求 |
|---|---|
| **综合命中率** | ≥ 85% |
| **样本量** | 近 3 个月同类任务 ≥ 10 个 |
| **无重大事故** | 近 3 个月无 P0 级 LL 触发记录 |
| **DL 覆盖度** | HAROLD-DECISION-LIBRARY 中同类有效条目 ≥ 3 条 |

### 5.3 DA 提议流程与时限

```
银月检测到某类型满足降级条件
    ↓
起草 DA 提议（使用下方 DA 模板）
    ↓
附证据包（命中率数据、典型案例、DL 引用清单）
    ↓
提交 Harold 审批（进入 MISSION_BOARD §四 待响应评审区）
    ↓
Harold 决策（时限 5 天）：
    ├─ ✅ 批准 → 更新 HAROLD-DELEGATION.md，添加授权记录
    ├─ 🔄 有条件批准 → 附加限制条件（如试运行 1 个月）
    └─ ❌ 拒绝 → 记录理由，3 个月后方可再次提议
```

**DA 提议被拒后的冷却期**：
- 首次被拒：3 个月后方可再次提议
- 连续两次被拒：6 个月后方可再次提议
- 需显著改善指标（命中率提升 10%+）方可突破冷却期

## 九、授权失效与恢复机制

### 9.1 自动失效触发条件

以下任一条件发生时，对应授权自动降为「⚠️ 暂停」，银月需在 24 小时内上报 Harold 决定是否恢复：

| 触发条件 | 说明 | 是否触发 LL |
|---|---|---|
| 执行出现重大偏差 | Deliverable 与 Harold 期望方向严重不符 | ✅ 是（P0） |
| Agent 引用 DL 错误 | Agent 错误引用了不适用的 Decision Library 条目，且连续发生 2 次 | ✅ 是（P1） |
| Harold 主动质疑 | Harold 对该类型的执行结果提出疑虑 | ⚠️ 视情况 |
| 所引用 DL 条目废止 | HAROLD-DECISION-LIBRARY 中支撑条目被标为「已废止」 | ❌ 否（直接失效） |
| 综合命中率跌破阈值 | 连续 3 个月命中率 < 80% | ✅ 是（P1） |

### 9.2 恢复流程

```
授权暂停
    ↓
银月 24h 内分析根因，提交 Harold
    ↓
Harold 决定：
    ├─ 恢复原授权（附加补充条件，如增加抽检比例）
    ├─ 降回上一级别（重新积累先例，3 个月后可再提议）
    └─ 永久废止（极端情况，需说明替代方案）
```
