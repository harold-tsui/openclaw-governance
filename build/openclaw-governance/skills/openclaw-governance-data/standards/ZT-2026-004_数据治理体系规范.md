## 〇、掌天数据文化宣言

> **"数据治理的本质，是给每一条信息一个家。"**
>
> 我们不依赖复杂的 API 网关，不依赖黑盒般的中间件。
> 我们回归最本质的**文档管理**--
> 每一个决策都有文档记录，每一条知识都有文件承载，每一个文件都在它该在的目录下。
>
> **可见，即可治。**

## 一、总则

### 1.1 适用范围

本规范管控掌天智能的所有数据源及流转过程：

* **协作平台**：飞书 (Feishu) -- *核心文档与协作流*
* **邮件通讯**：Gmail / 企业网页邮箱 -- *商务与外部联络流*
* **智能互联**：MCP (Model Context Protocol) -- *Agent 工具调用流*
* **网络情报**：Web 采集 / 浏览器插件 -- *外部信息流*
* **本地资产**：Local Files -- *私有化存储与归档*

### 1.2 核心原则（文档驱动版）

| 原则                        | 说明                                                                                       |
| --------------------------- | ------------------------------------------------------------------------------------------ |
| **单一真相源 (SSOT)** | 即使有多个副本，必须明确**唯一主文档**（Master Document）的位置                      |
| **文档即接口**        | Agent 之间不通过 API 调用交换数据，而是通过**读取/写入标准格式的文档**来协作         |
| **物理归档**          | 所有云端数据（飞书/邮件）最终必须在本地文件系统中有**物理备份**（Markdown/JSON/EML） |
| **路径语义化**        | 文件路径本身即是元数据，严禁随意存放                                                       |
| **隐私物理隔离**      | P0 数据严禁出现在共享文档目录中，必须物理隔离                                              |
| **PARA 对齐**         | 目录结构与 Projects / Areas / Resources / Archives 方法论严格对应                          |
| **矩阵制兼容**        | 同时支持职能线（Agent 个人）和项目线（跨 Agent 协作）两种协作模式                          |
| **最小权限**          | Agent 默认只能读写自己的工作目录；团队级和项目级需显式授权                                 |
| **全生命周期管理**    | 所有内容遵循 Draft → Active → Stale → Archived → Destroyed 五阶段流转                  |
| **隐私分级**          | 数据按敏感度分为 P0（绝密）/ P1（内部）/ P2（团队）/ P3（公开）四级                        |
| **证据链优先**        | 任何总结、结论、决策都必须能回溯到原始数据源                                               |
| **渐进式成熟**        | 遵循 DCMM 成熟度模型，先 L2 再 L3，不贪多冒进                                              |

## 四、数据架构（DCMM 域 3）

### 4.1 数据流转架构（基于文档管理）

我们不通过 API 服务层集成，而是通过**文件系统目录**进行集成。

```
┌─────────────────────────────────────────────────────────┐
│  Gold Layer · 知识库文档                                 │
│  Path: team/knowledge/ (Markdown + YAML Frontmatter)    │
│  ──────────────────────────────────────                 │
│  · 经过验证的最终知识，Agent 的"长期记忆"                 │
│  · 格式：标准 Markdown，必须包含 YAML 元数据头            │
│  · 来源：由 Silver 层文档人工/AI 整理合并而成             │
├─────────────────────────────────────────────────────────┤
│  Silver Layer · 标准化文档                               │
│  Path: local/processed/ (Markdown / JSON)               │
│  ──────────────────────────────────────                 │
│  · 清洗后的中间态数据                                    │
│  · 邮件摘要、飞书文档的 Markdown 导出版、MCP 数据的结构化记录 │
│  · 作用：去噪、提取关键信息、打标签                       │
├─────────────────────────────────────────────────────────┤
│  Bronze Layer · 原始归档文件                             │
│  Path: local/bronze/ (EML / HTML / Raw JSON)            │
│  ──────────────────────────────────────                 │
│  · 飞书：导出为 JSON/Markdown 快照                       │
│  · 邮件：原始 .eml 文件                                  │
│  · Web/MCP：原始 Response Payload (JSON/Text)           │
│  · 作用：法律证据、原始凭证，永久只读                     │
└─────────────────────────────────────────────────────────┘
```

### 4.2 数据分布：归属矩阵

| 数据源               | 原始格式 (Bronze) | 处理后格式 (Silver/Gold) | 主存位置 (Master)    | 本地归档位置             |
| -------------------- | ----------------- | ------------------------ | -------------------- | ------------------------ |
| **飞书文档**   | JSON Snapshot     | Markdown                 | 飞书云端             | `local/bronze/feishu/` |
| **Gmail/企邮** | .eml 文件         | Markdown 摘要            | 邮件服务器           | `local/bronze/email/`  |
| **Web 采集**   | HTML/PDF          | Markdown                 | 本地文件             | `local/bronze/web/`    |
| **MCP 数据**   | JSON              | YAML/Markdown            | 本地文件             | `local/bronze/mcp/`    |
| **P0 财务**    | Excel/PDF (加密)  | 仅摘要 (脱敏)            | **本地加密盘** | ❌ 严禁同步              |

### 4.3 目录结构规范（实体映射）

掌天智能的文件系统目录结构是数据架构的核心骨架，遵循 PARA (Projects/Areas/Resources/Archives) 方法论，并映射到三层数据湖仓（Bronze/Silver/Gold）。详细规范见引用文件：

- **引用**：ZT-2026-001 《掌天智能存储架构规范 v2.0》 （详见下文 Artifact: zt-directory-structure-v2）
- **引用**：ZT-2026-007 《交付物路径规范 v1.0》 （规范交付物存储路径与 Task-Card 管理）

此结构确保每份文档都有固定"家"，文件路径即体现元数据（如数据层级、状态、来源）。关键规则：

- **目录不可随意创建**：新子目录需 CDO 审批。
- **文件移动即流转**：例如，从 `00_Inbox/` 移到 `10_Projects/` 即表示从 Bronze 到 Silver 的升级。
- **隐私隔离**：P0/P1 数据仅限于特定目录（如 `secure/`），不得混入团队目录。
- **交付物路径强制化**：所有工作交付物必须存放于 `10_Projects/{Project-ID}/{Project-Topic}/deliverables/` 规范路径，禁止在 Agent 工作区长期存放（详见 ZT-2026-007）。

### 7.2 数据安全防护

#### 7.2.1 分级管控措施

| 级别         | 加密要求                           | 访问控制                   | 传输安全 | 备份策略               |
| ------------ | ---------------------------------- | -------------------------- | -------- | ---------------------- |
| **P0** | AES-256 加密，密钥存于离线 USB Key | 仅董事长（Agent 不可访问） | 不传输   | 本地加密备份，绝不上云 |
| **P1** | 文件系统权限 (chmod 600)           | 仅数据所属 Agent           | 本地传输 | 本地备份，不同步 CLOUD |
| **P2** | 无需文件级加密                     | 全体 Agent（按权限）       | HTTPS    | LOCAL + CLOUD 双备份   |
| **P3** | 无需                               | 所有人                     | 无限制   | 全渠道存档             |

#### 7.2.2 密钥管理方案

```
密钥分级存储：

P0 数据加密密钥
├── 主密钥（Master Key）     → 离线 USB Key，物理保管
├── 数据加密密钥（DEK）      → 由主密钥加密后存储于 LOCAL
└── 密钥轮换周期             → 每年 1 次

P1 访问控制
├── macOS Keychain          → 存储应用密码
├── 文件系统 ACL            → chmod 600 / chown
└── SSH 密钥                → Agent 间通信认证

云端加密（CLOUD）
├── 云平台自带加密           → 启用服务端加密
└── 独立加密密码             → 与本地密码不同
```

#### 7.2.3 "三不原则"（强制执行）

```
一、不上云：P0 数据，无论何种理由，永远不进入任何云服务
二、不共享：P1 数据，即使同团队 Agent，也不得主动推送
三、不留痕：涉敏操作必须写入审计日志，同时不在日志中记录敏感数据明文
```

### 7.3 数据安全审计

#### 7.3.1 审计日志五要素

每条审计日志必须包含：

```yaml
audit_log:
  who: "CDO"              # 谁执行了操作
  when: "2026-02-21T14:30:00+08:00" # 什么时间
  what: "read"                      # 什么操作 (read/write/delete/export/encrypt)
  where: "gold/persons/person_harold_xuzaihong"  # 操作了什么数据
  result: "success"                 # 操作结果 (success/denied/error)
  privacy_level: "P1"              # 被操作数据的隐私级别
  reason: "主数据更新"             # 操作原因（P0/P1 操作必填）
```

#### 7.3.2 审计检查机制

| 审计活动    | 频率               | 负责人 | 检查内容 |
| ----------- | ------------------ | ------ | -------- |
| P0 访问审计 | 每次发生时实时告警 |        |          |
