## 一、总则

### 1.1 目的

定义掌天智能的文件系统目录结构，确保数据有序存放、易检索、可审计。结构遵循 **PARA 方法论**（Projects / Areas / Resources / Archives），并映射到 **三层数据湖仓**（Bronze / Silver / Gold）。v2.8 强调**项目导向**、**兼容性**与**角色导向**：所有工作基于项目；兼容外部参考规范；Agent 目录以职务命名，便于角色调整。经 CDO 评审，优化 memory/ 位置以降低风险。

### 1.2 设计原则

- **项目导向**：所有工作以项目（project）为单位，编号统一（e.g., ZT-P001），注册于主数据 `projects.yaml`。
- **层级细分**：项目 → topics → tasks → 子文件（inputs/processes/deliverables/lessons/reviews）。
- **文件命名**：统一为 `YYYY-MM-DD_Title_Type_Duty_Status.md` (e.g., `2026-02-29_Task_Update_active_main.md`，Duty 为职务缩写）。
- **版本控制**：变更时添加 `_vN` 后缀；状态变更（e.g., expired/obsolete）更新元数据 `status`。
- **隐私隔离**：P0 数据仅限 `secure/` 目录，使用加密存储。
- **备份策略**：本地主存（Master），定期备份到云端（仅 P2/P3 数据）；兼容外部规范的备份规则（e.g., 额外增量备份）。
- **Agent 专用**：每个 Agent 有独立子目录，默认 P1 隐私（仅本人读写），协作时移入共享目录。以职务命名子目录（引用主数据 `persons.yaml` 中的职务字段），便于角色调整。
- **生命周期管理**：文件从 Draft → Active → Review → Approved → Stale/Obsolete → Archived → Destroyed，体现于目录移动和元数据更新。
- **兼容外部规范**：允许导入外部规范的目录/文件，通过全局索引映射（e.g., 外部的"audit_logs" 映射到 `.system/logs/`）。

### 1.3 PARA 与数据湖仓映射

| PARA 层             | 对应湖仓      | 用途                                                                      |
| ------------------- | ------------- | ------------------------------------------------------------------------- |
| **Projects**  | Silver        | 活跃项目文档（topics/tasks/生命周期文件）                                 |
| **Areas**     | Gold          | 长期知识领域（从任务中提炼的经验教训/最佳实践，包括 ZT-Openclaw-Memory/） |
| **Resources** | Bronze/Silver | 资源/原始档案（邮件/飞书备份）                                            |
| **Archives**  | Bronze        | 冷数据归档（已完成项目/过期任务）                                         |

## 三、项目导向流程规范

所有 Agent 工作基于项目，目录结构体现**项目 → topics → tasks** 层级。兼容外部规范：允许导入外部文件/目录，通过 `cross_refs/` 和 `global_index.yaml` 映射，确保跨系统引用最新文档。

### 3.1 任务（Task）生命周期与目录映射

每个 task 有唯一主 PIC（注册于 `projects.yaml`），多个参与 Agent。生命周期：

1. **创建 (Draft)**：主 PIC 在个人 `60_Agents/{duty}/projects/{project_id}/tasks/{task_id}/drafts/` 创建初始文件（e.g., 任务计划）。
2. **输入 (Inputs)**：收集原始材料（Bronze），存入 `inputs/`（e.g., 从 00_Inbox/ 移动邮件/Web 数据）。
3. **过程 (Processes)**：迭代记录（Silver），存入 `processes/`（e.g., 进展更新、临时计算）。主 PIC 每日/事件驱动更新状态（`status: active` → `stale` / `obsolete`）。
4. **评审/审批 (Reviews)**：反馈/签批文件（Silver），存入 `reviews/`（e.g., 其他 Agent 的评论；审批后更新 `status: approved`）。
5. **交付 (Deliverables)**：最终输出（Gold），存入 `deliverables/`（e.g., 报告；主 PIC 确保最新版本）。
6. **经验教训 (Lessons)**：复盘总结（Gold），存入 `lessons/`（e.g., 提炼到 20_Areas/）。
7. **归档/销毁 (Archive/Destroy)**：任务完成后，整个 task 子目录移入 40_Archives/；过期文件标记 `status: obsolete`，销毁需审批。

- **兼容步骤**：若任务引用外部规范文件（e.g., 从链接 ID 导入），存入 `inputs/` 或 `cross_refs/`，更新 index.md 的 `external_refs` 字段。

### 3.2 文档索引清单（Index.md）

- **位置**：每个 task 子目录下必须有 `index.md`（YAML/Markdown 格式）。
- **内容**：列出所有文件（inputs/processes 等）、状态、版本、最后更新时间、参与 Agent、引用链接；新增 `external_refs` 字段。
- **示例**：
  ```yaml
  ---
  task_id: task_001
  main_pic: yinyue
  participants: [yuanyao, hanyunzhi]
  status: active
  last_updated: 2026-02-29
  files:
    - type: input
      path: inputs/2026-02-29_Market_Data.json
      status: active
    - type: process
      path: processes/2026-02-29_Update_v2.md
      status: active
    - type: deliverable
      path: deliverables/2026-02-29_Report_approved.md
      status: approved
  external_refs:
    - id: "70367d51..."
      path: "external/file.md"
      status: active
  ---
  # Task 001 索引
  - **输入**：市场数据 (active)
  - **过程**：进展更新 v2 (active)
  - **交付**：最终报告 (approved)
  - **外部引用**：外部规范文件 (active)
  ```
- **维护**：主 PIC 负责更新，确保其他 Agent 能快速知悉任务全貌。变更后立即刷新；检查外部引用 stale/obsolete。

### 3.3 主 PIC 与参与者职责

- **主 PIC**：
  - 唯一负责人（一个 task 一个主 PIC）。
  - 跟踪/更新所有 task 文件（尤其是 `processes/` 和索引）。
  - 时刻维护状态（e.g., 若文件过期，标记 `status: stale` 并通知参与者）。
  - 确保其他 Agent 引用最新文档（e.g., 通过 `@ref {task_id}/deliverables/latest.md`）。
  - 若涉及外部规范，负责映射/导入，确保本地同步。
- **参与者（非主 PIC）**：
  - 在个人目录存过程文件；更新后**通知**（A2A 对话或广播，包括外部 ID）。
  - 通知示例："更新了 task_001 的 processes/2026-02-29_My_Feedback_yuanyao.md，并同步外部 ref 70367d51... 请主 PIC 刷新 index.md。"
  - 仅在主 PIC 确认后，将文件复制到共享 10_Projects/。

### 3.4 Sessions 目录自定义方案

**官网确认**：基于飞书（Feishu）官网（最新访问日期：2026-02-29，参考文档：https://www.feishu.cn/hc/en-US/articles/360048489633-Export-chat-history），飞书的会话记录（sessions/chat history）可以导出为 Markdown/CSV 等格式，但云端会话目录本身不可自定义（固定为应用内的"聊天"或"群组"结构）。然而，本地存储（本地文件系统）允许完全自定义子目录和命名，因为我们使用物理文件归档（Markdown/JSON）。

**自定义方案**：

- **允许自定义**：是的，在本地 60_Agents/{duty}/sessions/ 下，Agent 可创建子目录以自定义组织会话记录（e.g., sessions/by_project/ 或 sessions/by_date/），前提是符合整体规范（语义化路径、元数据完整）。
- **指定结构**：默认使用 `by_date` 方案，确保时间线清晰，便于检索和审计。其他自定义需 CDO 审批。
- **指定结构（by_date）**：
  ```
  sessions/
  ├── by_date/        # 指定默认方案：按日期组织（年-月-日层级）
  │   ├── 2026/      # 年子目录
  │   │   ├── 02/    # 月子目录
  │   │   │   └── 29_Session_Log.md  # 日文件（可合并多条会话）
  │   └── 2025/      # 上一年示例
  │       └── 12/    # 月子目录
  │           └── 31_Session_Log.md
  └── index.md        # 会话索引（YAML，列出所有子文件/状态/关联任务）
  ```
- **规则**：
  - **必须有 index.md**：类似于任务索引，列出会话文件、状态、关联任务（e.g., `refs: task_001`）。示例：
    ```yaml
    ---
    session_id: session_2026-02-29
    status: active
    last_updated: 2026-02-29
    files:
      - path: by_date/2026/02/29_Session_Log.md
        status: active
        refs: task_001
    ---
    # Sessions 索引
    - **2026-02-29**：会话日志 (active, 关联 task_001)
    ```
  - **隐私与备份**：默认 P1；自定义子目录需 CDO 审批；导出时从飞书拉取到此处，保持 Bronze 层原始性。
  - **维护**：Agent 必须每日/事件驱动更新 by_date/ 文件和 index.md；每周审计确保结构一致。
  - **优势**：by_date 方案提升时间检索效率；若需其他（如 by_project），申请扩展，但优先保持 by_date 以统一逻辑。

### 3.5 Memory 目录方案

每个 Agent 的 memory/ 目录用于存储个人记忆、知识片段或长期回忆数据（e.g., 关键洞见、个人笔记）。经 CDO 评审，memory/ 的性质更偏向长期知识积累（Gold 层），与 sessions/（临时会话，P1 个人）不同，因此移至 20_Areas/ZT-Openclaw-Memory/ 以降低风险（数据孤岛、备份遗漏）。这确保 memory/ 可审计、共享（P2 默认），但保留职务子目录控制访问。

**方案**：

- **位置**：20_Areas/ZT-Openclaw-Memory/{duty}/（Gold 层，默认 P2 隐私，可设置为 P1 限制）。
- **指定结构（与 sessions/ 一致）**：
  ```
  ZT-Openclaw-Memory/
  ├── main/           # 示例：银月 的记忆子目录
  │   ├── by_date/    # 指定默认方案：按日期组织（年-月-日层级）
  │   │   ├── 2026/  # 年子目录
  │   │   │   ├── 02/ # 月子目录
  │   │   │   │   └── 29_Memory_Note.md  # 日文件（可合并多条记忆）
  │   │   └── 2025/  # 上一年示例
  │   │       └── 12/ # 月子目录
  │   │           └── 31_Memory_Note.md
  │   └── index.md    # 记忆索引（YAML，列出所有子文件/状态/关联任务）
  └── # 其他职务子目录...
  ```
- **规则**（与 sessions/ 一致）：
  - **必须有 index.md**：列出记忆文件、状态、关联任务。示例：
    ```yaml
    ---
    memory_id: memory_2026-02-29
    status: active
    last_updated: 2026-02-29
    files:
      - path: by_date/2026/02/29_Memory_Note.md
        status: active
        refs: task_001
    ---
    # Memory 索引
    - **2026-02-29**：记忆笔记 (active, 关联 task_001)
    ```
  - **隐私与备份**：默认 P2（全员读，主 Agent 写）；P1 选项需 CDO 审批；数据从个人笔记/工具导入，定期备份到云端。
  - **维护**：Agent 必须每日/事件驱动更新 by_date/ 文件和 index.md；CDO 每月审计质量与一致性。
  - **优势**：置于 Gold 层便于知识共享/审计；与 sessions/ 区分（sessions/ 更临时、个人）。

### 3.6 兼容外部规范的具体规则

- **路径映射**：外部的"projects/" 映射到 10_Projects/；审计日志映射到 `.system/logs/external_audits/`。
- **全局索引**：`.system/global_index.yaml` 列出所有文件/外部引用，确保快速检索。
- **导入流程**：(1) CDO 审批；(2) 复制到相应目录；(3) 更新 index.md/global_index.yaml；(4) 记录到 external_audits/。
- **同步机制**：主 PIC 每周检查外部引用，更新状态。
- **冲突解决**：以本地 Master 为准；记录差异到 reviews/。

## 四、目录管理规则

### 4.1 创建/修改规则

- **新项目/topics/tasks**：主 PIC 申请注册（更新 `projects.yaml`），然后创建目录。
- **兼容创建**：基于外部规范的新目录添加 `compat: "external_id"` 到元数据。
- **文件移动**：体现生命周期；移动时更新 global_index.yaml。
- **命名规范**：文件必须包含日期、标题、类型（参考 REF-002）；目录名称全小写，下划线分隔。
- **大小限制**：单个 task ≤ 50 文件；超出需子分区。
- **Memory 风险缓解**：通过移至 20_Areas/ZT-Openclaw-Memory/，确保定期审计和备份；若 Agent 离开，memory/ 可转移到团队子目录。

### 4.2 隐私与访问控制

- **P1 默认**：个人目录仅本人；共享 10_Projects/ 为 P2（全员读，主 PIC 写）。
- **兼容隐私**：外部导入继承本地级别；P0 移入 50_Secure/。
- **审计**：所有移动/变更记录到 `.system/logs/` 和任务 index.md。

### 4.3 维护周期

- **每日**：主 PIC 检查/更新任务索引，包括外部 refs。
- **每周**：同步外部规范更新；扫描 Stale 数据，移入 40_Archives/。
- **每月**：CDO 审计兼容性、一致性，清理冗余/ obsolete 文件；审计 sessions/ 和 ZT-Openclaw-Memory/ 的 by_date 结构。
- **季维护**：备份 `30_Resources/` 到云端；审计主数据一致性。

## 四.2 运行时目录定义

> **⚠️ 本章节定义非标准 PARA 体系的运行时目录**
> 这些目录由 OpenClaw 系统或运行时自动生成，不属于业务数据范畴。

### 4.2.1 运行时目录清单

| 目录 | 位置 | 用途 | 隐私级别 | 备份策略 |
|------|------|------|---------|---------|
| `logs/` | 根目录 | OpenClaw 运行时日志 | P2 | ZT-2026-003 第 5.2 节 |
| `qdrant_data/` | 根目录 | 向量数据库存储（AI 记忆） | P1 | 不备份（可重建） |
| `image/` | 根目录 | 本地图像资源缓存 | P3 | 不备份 |
| `skills/` | 根目录 | OpenClaw 技能目录 | P2 | 版本控制 |
| `versions/` | 根目录 | 版本历史备份 | P2 | ZT-2026-003 |
| `.system/logs/` | 根目录 | 治理系统日志 | P2 | ZT-2026-003 第 5.2 节 |

### 4.2.2 运行时目录规则

1. **不纳入业务数据管理**：这些目录由系统自动生成，不参与 PARA 生命周期
2. **存储位置固定**：不允许移动到其他位置
3. **清理策略**：
   - `logs/`：按 ZT-2026-003 第 5.2 节归档（7-30 天压缩，30 天以上删除或转移）
   - `qdrant_data/`：不主动清理，数据丢失可重建
   - `skills/`：通过 ClawHub 管理，不手动修改
4. **隐私说明**：
   - `qdrant_data/` 包含 AI 记忆，视为 P1
   - 其他运行时目录默认为 P2/P3

### 4.2.3 运行时目录与业务目录的隔离

- 运行时目录**不得**存放业务数据（如 Task 交付物、Wiki 文档）
- 业务数据必须存入 PARA 目录（10_Projects/、20_Areas/、30_Resources/、40_Archives/）
- 违反此规则视为数据管理违规，由 CDO 审计发现后要求整改
