## 一、设计理念

### 1.1 核心原则

**"Skill 只关注执行"** - Harold Tsui (2026-04-03)

这个原则的本质是**运行时认知负载优化**：
- Agent 执行时只需要"怎么做"（How）
- 人类/开发者需要"为什么"（Why）
- 混合两者会导致 token 浪费和决策干扰

### 1.2 分离标准

**关键判断标准**：删除该内容后，Agent 执行时是否会做出错误决策？
- 是 → 留在 `SKILL.md`
- 否 → 移到 `DESIGN.md` 或 `README.md`

### 1.3 文档分层策略

```
执行频率分层（方案 C）：
├── SKILL.md        # 热路径：100% 执行必需（Phase 序列 + 屏障规则）
├── SKILL-EDGE.md   # 温路径：~20% 触发（失败矩阵扩展 + 组合场景）
├── README.md       # 人类文档：架构图、快速上手
└── DESIGN.md       # 冷路径：设计理念、演进记录（调试/审查用）
```

**分层逻辑**：
- 热路径：每次执行必读
- 温路径：遇到异常/边界情况时读取
- 冷路径：理解设计意图、调试时读取

## 二、Phase 屏障设计

### 2.1 为什么需要四种屏障类型？

| 屏障类型 | 设计原因 | 不这样做的后果 |
|----------|----------|----------------|
| **硬屏障** (Phase 1→2A) | 基础层必须完全就绪，否则后续模块无法正确初始化 | 部分模块在依赖未就绪时启动，导致运行时错误 |
| **硬屏障 + 状态广播** (Phase 2A→2B) | 2B 模块需要根据 2A 健康状态调整初始化策略 | quality 不知道 data 是 degraded，尝试写操作导致失败 |
| **条件屏障** (Phase 2B→3) | 某些失败组合会使得 Phase 3 无意义或危险 | hierarchy failed 时 task 无法创建项目，但仍加载浪费资源 |
| **软屏障** (Phase 3→4) | Phase 4 是增强功能，不应阻塞主流程 | infrastructure 失败导致整个系统无法使用，过度降级 |

### 2.2 状态广播机制详解

**问题背景**：
- Phase 2A 和 2B 是并行加载
- 2B 模块（quality, delegation）依赖 2A 模块（data, hierarchy）的状态
- 但 2A 和 2B 是同时启动的，2B 无法直接读取 2A 的运行时状态

**解决方案**：
```
Phase 2A 完成时：
  1. 收集所有 2A 模块的最终状态
     - hierarchy_status: success|failed
     - data_status: success|degraded|failed
  2. 广播状态快照到共享上下文
  3. 通知 2B 模块"状态已就绪"

Phase 2B 初始化时：
  1. 等待状态广播完成（屏障）
  2. 读取状态快照
  3. 根据快照调整初始化策略：
     - quality: 如 data_status==degraded，进入只读审核模式
     - delegation: 如 hierarchy_status==failed，标记自身为 failed
```

**实现要点**：
- 状态快照必须是**不可变**的（避免竞态条件）
- 2B 模块必须**显式等待**屏障信号（不能轮询）
- 状态广播必须**原子性**（要么全有，要么全无）

## 三、Tier/Phase 映射关系（P1-3 统一）⭐

### 3.1 ARCH-001 Tier 分层 vs 当前 Phase 实现

**背景**：ARCH-001 v2.1 使用 Tier 1/2/3 分层（静态架构），当前实现使用 Phase 1-4 启动序列（动态加载）。为避免认知摩擦，明确映射关系如下：

```
ARCH-001 Tier（静态架构） → 当前 Phase 实现（动态启动）

Tier 1（核心基础层）
├─ 定义：系统运行必需的基础服务
└─ 对应 Phase: Phase 1 + Phase 2A + Phase 2B
    - Phase 1: core, config, dispatch
    - Phase 2A: hierarchy, data
    - Phase 2B: quality, delegation

Tier 2（执行层）
├─ 定义：任务执行和系统巡检
└─ 对应 Phase: Phase 3
    - Phase 3: task, heartbeat

Tier 3（增强层）
├─ 定义：按需加载的扩展能力
└─ 对应 Phase: Phase 4 + L3* 惰性加载
    - Phase 4: infrastructure, evolution
    - L3*: intelligence, content, visual
```

### 3.2 映射对照表

| ARCH-001 Tier | 层级命名 | Phase 序列 | Skills | 加载时机 |
|---------------|----------|------------|--------|----------|
| **Tier 1** | 核心基础层 | Phase 1-2B | core, config, dispatch, hierarchy, data, quality, delegation | 启动触发 |
| **Tier 2** | 执行层 | Phase 3 | task, heartbeat | 意图/定时触发 |
| **Tier 3** | 增强层 | Phase 4 + L3* | infrastructure, evolution, intelligence, content, visual | 显式/按需触发 |

### 3.3 为什么需要两套语言？

**Tier 分层（ARCH-001）**：
- 用途：描述**静态架构**（"是什么"）
- 优势：清晰的层级依赖关系，便于理解系统结构
- 场景：架构设计、Skill 分类、新 Agent 培训

**Phase 序列（当前实现）**：
- 用途：描述**动态启动**（"什么时候加载"）
- 优势：明确的加载顺序和屏障规则，便于执行
- 场景：Agent 启动、失败处理、性能优化

**映射原则**：
- Tier 1 可能跨越多个 Phase（如 delegation 是 Tier 1，但在 Phase 2B 加载，因为依赖 hierarchy）
- Phase 序列是 Tier 分层的时间展开
- 选择使用哪套语言取决于场景：架构讨论用 Tier，执行流程用 Phase

## 四、失败矩阵扩展

### 4.1 单点失败的原因分析

| 失败节点 | 为什么这样处理？ | 替代方案及问题 |
|----------|------------------|----------------|
| **core 失败 → 终止** | core 是根依赖，无降级可能 | 尝试降级 → 所有模块都会失败，不如早终止 |
| **config 失败 → 降级** | config 只有默认值，可用内置 defaults | 终止 → 过度反应，系统仍可运行 |
| **dispatch 失败 → 终止** | 无法路由意图，系统失去核心功能 | 降级 → 无法实现，没有替代路由机制 |
| **hierarchy 失败 → 严重降级** | task 需要项目/Topic 信息 | 继续 → task 创建会失败，产生更多错误 |
| **data 失败 → 只读模式** | data 提供路径校验，只读操作不需要 | 终止 → 过度反应，审核仍可进行 |
| **quality 失败 → 轻度降级** | quality 只影响闭环验证 | 终止 → 过度反应，task 仍可创建和执行 |
| **delegation 失败 → 阻止 task** | delegation 提供权限验证 | 继续 → 安全风险，未授权操作可能发生 |

### 4.2 组合失败场景分析

#### 场景 1：quality + delegation 同时失败

**决策**：等同于 delegation 失败（阻止 task 加载）

**原因**：
- delegation 失败 = 无法验证权限 = 安全风险
- quality 失败 = 无法验证闭环 = 质量问题
- 安全风险 > 质量问题（安全优先原则）

**替代方案**：
- 方案 A：继续加载 task，但记录警告
  - 问题：未授权操作可能发生，违反安全原则
- 方案 B：只阻止需要权限验证的 task
  - 问题：实现复杂，边界模糊，容易出错

#### 场景 2：hierarchy + data 同时失败

**决策**：等同于 hierarchy 失败（阻止 Phase 3）

**原因**：
- hierarchy 失败 → task 无法创建项目/Topic
- data 失败 → task 无法校验路径
- 两者同时失败 → task 几乎无法执行任何操作
- 继续加载 Phase 3 只会产生更多错误

#### 场景 3：config + data 同时失败

**决策**：降级到 defaults + 只读模式，继续运行

**原因**：
- config 失败 → 使用内置 defaults（功能受限）
- data 失败 → 只读模式（无法写操作）
- 但核心功能（意图路由、任务创建）仍可用
- 用户仍可工作，只是部分功能受限

### 4.3 未覆盖的组合失败场景

**已知未覆盖场景**：
1. config + dispatch 同时失败
   - ~~当前策略：未定义~~ → **已解决 ✅**
   - 分析：dispatch 是 Phase 1 致命模块，失败直接 TERMINATE，不进入 Phase 2+ 组合处理。config 失败是否同时发生无意义，结果同 dispatch 单点失败。
   - 已补充到 `SKILL.md` §4.2 矩阵（标注 Phase 1 TERMINATE，N/A Phase 3）
2. quality + data 同时失败
   - ~~当前策略：未定义~~ → **已覆盖 ✅**（`SKILL.md` §4.2 矩阵中已有 `quality + data` 行：data 优先传播，降级继续，带警告）
3. infrastructure + evolution 同时失败（Phase 4）
   - 当前策略：向用户报告"增强功能暂时不可用"
   - 已覆盖 ✅

## 六、设计决策记录

### 决策 1：为什么 task 显式依赖 data？（v5.9.0）

**背景**：之前 task 隐式依赖 data，v5.9.0 补全显式声明

**原因**：
- task 创建时需要校验路径（data 的职责）
- 隐式依赖导致调试困难（不知道 task 为什么失败）
- 显式声明后，data 失败时 task 可以提前知道并降级

**替代方案**：
- 保持隐式依赖
  - 问题：task 运行时才发现 data 不可用，产生 confusing 错误

### 决策 2：为什么 Phase 4 是惰性加载？（v5.9.0）

**背景**：Phase 4（infrastructure, evolution）只在用户明确请求时加载

**原因**：
- Phase 4 是增强功能，不是核心功能
- 惰性加载节省 token 和启动时间
- 失败不影响主流程（软屏障）

**替代方案**：
- Phase 4 在启动时加载
  - 问题：浪费 token（80% 会话用不到 Phase 4）
  - 问题：infrastructure 失败会阻塞整个系统

### 决策 3：为什么 delegation 失败阻止 task，但 quality 失败不阻止？（v5.9.0）

**背景**：两者都是 L3 规范层，但失败处理不同

**原因**：
- delegation 失败 → 无法验证权限 → 安全风险
- quality 失败 → 无法验证闭环 → 质量问题
- 安全 > 质量（安全优先原则）

**替代方案**：
- 两者都继续
  - 问题：delegation 失败时，未授权操作可能发生
- 两者都阻止
  - 问题：quality 失败时，task 仍可安全执行（只是无法闭环验证）

## 七、未来改进方向

### 7.1 方案 C：语义分层 + 渐进加载

**当前状态**：待评估 Agent 元认知能力后决定

**方案内容**：
```
skills/openclaw-governance-core/
├── SKILL.md        # 热路径：Phase 序列 + 屏障规则（100% 必需）
├── SKILL-EDGE.md   # 温路径：失败矩阵扩展 + 组合场景（约 20% 触发）
├── README.md       # 人类文档：架构图、快速上手
└── DESIGN.md       # 冷路径：设计理念、演进记录
```

**前提条件**：
- Agent 具备"判断何时需要查阅 DESIGN.md"的元认知能力
- 如果 Agent 不具备，反而会出现信息缺失导致的执行错误

**评估方法**：
1. 设计测试集（包含 10 个复杂边界场景）
2. 让 Agent 在只有 SKILL.md 的情况下执行
3. 统计 Agent 主动请求"我需要更多信息"的次数
4. 如果 > 70%，说明具备元认知能力，可实施方案 C

### 7.2 文档有效性验证机制

**目标**：每次 SKILL.md 修改后，验证 Agent 执行正确率

**方法**：
1. 建立失败场景测试集（20 个场景）
2. 每次修改 SKILL.md 后，运行测试集
3. 统计通过率
4. 如果 < 95%，说明压缩过度，需要恢复部分内容

**实施优先级**：中（在方案 C 之后）
