## ⚠️ 更新时机说明（v3.0 重写）

**MISSION_BOARD 是派生视图，§一 状态总览的数字从各 section 计数得出。**
**各 section（§二~§六）是手工维护的协调视图，§一 是自动派生的。**

### 更新规则

| 触发事件 | 更新内容 | 时限 |
|----------|----------|------|
| **新 Task 创建** | §五 待处理队列 + §一 派生刷新 | 立即 |
| **Task 状态变更** | §三/§六 + §一 派生刷新 | 立即 |
| **Topic/Project 状态变更** | §一.2 四层跟踪概览 | 立即 |
| **Heartbeat PDCA Act** | §一 派生刷新 + §九 时间戳 | 每次 Act |
| **交付物提交** | §四 待响应评审区 + §一 派生刷新 | 立即 |
| **阻塞发生/解除** | §二 阻塞区 + §一 派生刷新 | 立即 |

### 派生刷新算法（v3.0 新增）

§一 状态总览的每个指标都有明确的派生算法，不需要手写：

| 指标 | 派生算法 |
|------|---------|
| 活跃任务数 | §三 进行中区 的 Task ID 数（含 Task ID 的表格行） |
| 待处理任务数 | §五 待处理队列 的 Task ID 数（含 Task ID 的表格行） |
| 阻塞任务数 | §二 阻塞区 的 Task ID 数（含 Task ID 的表格行） |
| 本周完成任务数 | §六 已完成区 中 `close_date` 在本周范围内的 Task ID 数 |
| 今日 Harold Review | §四 待响应评审区 中 Harold 为待审批人的 Task ID 数 |

**"本周"定义**：周一 00:00 ~ 周日 23:59。每周一首次 heartbeat 时，§六 已完成区清零（上周记录归档至 `memory/mission-board-archive/`）。

### 禁止行为

- ❌ 手写 §一 状态总览的数字（必须从各 section 计数派生）
- ❌ 批量更新（滞后更新）
- ❌ 只在 Task 完成时更新
- ❌ Heartbeat 执行但不刷新 §一 派生指标

## 状态标记速查

> **状态标记定义与流转规则**：详细定义见 governance-task v6.0.1 §1.4 + TMPL-TASK-CARD v3.1 §七 7.1

| 标记 | 含义 | 默认 P 级 | 银月动作 | 流转条件 |
|---|---|---|---|---|
| `[ ]` | 待接收 | — | 正常跟踪；等待 Task PIC 接收 | Task 创建 / 移入 §五 |
| `[P]` | 执行中 | — | 正常跟踪；推进阻塞任务 | Task PIC 接收后 |
| `[V]` | 待验收 | 继承原 Task P 级 | 立即启动知识沉淀，推进至 `[x]` | 交付物提交后 |
| `[x]` | 已完成 | — | 文件移入 40_Archives（如需归档） | 验收通过后 |
| `[?]` | 阻塞 | 升一级 | 识别阻塞原因，尝试解除或升级 | 依赖未满足 / 等待外部输入 |
| `[!]` | Harold 待决 | 升一级 | 确认已移入 §四，24h 超时提醒 | 需要 Harold 决策 |

**状态流转链**：`[ ] → [P] → [V] → [x]`

**§四 内部流转**：
```
任务移入 §四（状态 `[!]`）
    ↓ 银月按 Heartbeat 周期 follow up
Harold 明确 approve
    ↓
标记 `[V]`，移出 §四 → 进入知识沉淀流程
```

**Issue Ticket 子状态**（在 `[P]` 内部细化）：
`[A]` 分析中 → `[F]` 修复中 → `[T]` 待测试 → `[TT]` 测试中 → `[Q]` 待抽查 → `[QQ]` 抽查中 → `[C]` 待关闭

**特殊情况**：
- 知识沉淀进行中：不使用 `[~]` 状态，改为 Zone C 后处理（3 工作日宽限期）
- 归档完成：使用 `[archived]` 状态（由 archive_task() 触发）
