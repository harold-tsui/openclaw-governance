# INCIDENT-POSTMORTEM-SOP · 故障复盘流程

> **版本**：v1.0
> **触发条件**：SEV-1/SEV-2 故障恢复后
> **审批级别**：L3（Harold 审核）
> **预计耗时**：2-4 小时（复盘会议 + 文档撰写）

---

## 一、流程概览

```
Step 1: 准备复盘 → Step 2: 召开复盘会议 → Step 3: 根因分析 → Step 4: 制定改进措施 → Step 5: 撰写复盘报告 → Step 6: 审核发布
```

---

## 二、复盘触发条件

| 级别 | 复盘要求 | 复盘截止时间 |
|------|----------|--------------|
| **SEV-1** | **必须复盘** | 恢复后 24 小时内 |
| **SEV-2** | **建议复盘** | 恢复后 72 小时内 |
| **SEV-3** | 可选复盘 | 恢复后 1 周内 |
| **SEV-4** | 不复盘 | - |

---

## 三、详细步骤

### Step 1: 准备复盘（预计 30min）

```
□ 收集故障资料
  - 时间线
  - 相关日志
  - 沟通记录
  - 影响数据
□ 确定复盘参与人
□ 安排复盘会议
□ 发送会议邀请
□ 输出：复盘准备完成
```

**准备材料**：
- 故障时间线（精确到分钟）
- 影响分析报告
- 恢复操作记录
- 相关日志和截图
- 用户反馈

---

### Step 2: 召开复盘会议（预计 1-2 小时）

```
□ 故障概述（10min）
□ 时间线回顾（15min）
□ 影响分析（15min）
□ 根因分析（30min）
□ 改进措施讨论（30min）
□ 行动项确认（10min）
□ 输出：会议纪要
```

**会议原则**：
- **无责原则**：不追究个人责任
- **聚焦改进**：关注系统性改进
- **全员参与**：鼓励所有人发言
- **记录完整**：记录所有想法

**会议参与人**：
- 故障指挥人
- 响应团队成员
- 相关 Agent
- 银月（协调）
- Harold（SEV-1 必须）

---

### Step 3: 根因分析（预计 30min）

```
□ 使用 5 Whys 方法
□ 追问至少 5 层原因
□ 找到根本原因
□ 确认非表面原因
□ 输出：根因分析结果
```

**5 Whys 示例**：

```
问题: 服务不可用

Why 1: 数据库连接失败
Why 2: 数据库连接池耗尽
Why 3: 连接未正确释放
Why 4: 异常处理逻辑缺失
Why 5: 代码审查未覆盖异常场景

根因: 缺乏异常场景代码审查机制
```

**其他分析方法**：
- 鱼骨图（因果分析）
- 故障树分析
- 时间线分析

---

### Step 4: 制定改进措施（预计 30min）

```
□ 针对每个根因制定改进措施
□ 确定责任人和截止时间
□ 评估措施可行性
□ 确认优先级
□ 输出：改进措施清单
```

**改进措施分类**：

| 类别 | 说明 | 示例 |
|------|------|------|
| **立即修复** | 防止再次发生 | 代码修复 |
| **流程改进** | 改进工作流程 | 增加审查环节 |
| **监控增强** | 提前发现问题 | 增加告警 |
| **文档更新** | 沉淀经验 | 更新 SOP |
| **培训学习** | 提升能力 | 分享会 |

**措施格式**：
```markdown
| 措施 | 责任人 | 截止时间 | 优先级 |
|------|--------|----------|--------|
| {action} | {owner} | {due_date} | P1/P2/P3 |
```

---

### Step 5: 撰写复盘报告（预计 1 小时）

```
□ 整理会议纪要
□ 撰写复盘报告
□ 附上时间线和日志
□ 列出改进措施
□ 输出：复盘报告
```

**报告结构**：
1. 概述
2. 时间线
3. 影响分析
4. 根因分析
5. 改进措施
6. 经验总结
7. 附录

---

### Step 6: 审核发布（预计 30min）

```
□ 银月初审
□ Harold 终审（SEV-1）
□ 发布复盘报告
□ 归档到故障记录
□ 跟踪改进措施
```

**审核要点**：
- 根因分析是否深入
- 改进措施是否可行
- 责任人和时间是否明确
- 经验是否沉淀
