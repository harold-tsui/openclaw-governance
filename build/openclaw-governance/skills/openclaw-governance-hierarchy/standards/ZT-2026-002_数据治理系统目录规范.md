## 一、设计原则

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **单一真相来源** | 每个文件只有一个权威位置，禁止重复存放 |
| **层级清晰** | 按照 L1-L6 层级组织，职责分明 |
| **动静分离** | 静态规范文件和动态日志文件分开管理 |
| **易于查找** | 目录结构直观，文件命名统一 |
| **向前兼容** | 保留历史版本，支持追溯审计 |

### 1.2 变更说明

**v2.3 vs v2.2**：
| 变更项 | v2.2 | v2.3 | 原因 |
|--------|------|------|------|
| 系统级目录结构 | 分散在 current/ 下 | 统一放到 projects/ 目录 | 更清晰的组织结构 |
| 项目 ID 命名 | SYSTEM-OPS, AGENT-RESOURCE | SYS-OPS, SYS-AGENT | 统一 SYS-* 前缀 |
| 新增 SYS-DATA | 无 | 新增 sys-data/ 目录 | 完善数据治理体系 |
| yaml 双保险 | 仅目录存在 | 新增 dir_path 字段 | 确保一致性 |

**v2.2 vs v2.1**：
| 变更项 | v2.1 | v2.2 | 原因 |
|--------|------|------|------|
| 系统级目录 | 未定义 | 新增 projects/ 统一管理系统级项目 | 正式定义系统级目录结构 |
| 目录结构 | 不含系统级目录 | 新增 4.10 章节详细定义 | 完善目录规范 |

**v2.1 vs v2.0**：
| 变更项 | v2.0 | v2.1 | 原因 |
|--------|------|------|------|
| 备份目录 | 未定义 | 新增 `${OPENCLAW_BACKUP_WORKSPACE}` | 统一备份管理 |
| 备份命名 | 未定义 | `原文件(夹)+.bak.日期` | 规范备份命名 |

**v2.0 vs v1.0**：
| 变更项 | v1.0 | v2.0 | 原因 |
|--------|------|------|------|
| HAROLD-DECISION-LIBRARY | 同时存在于 policies/ 和 decisions/ | 仅存在于 decisions/ | 消除重复，明确归属 |
| heartbeat-logs 位置 | governance/heartbeat-logs/ | governance/logs/heartbeat/ | 统一日志管理 |
| audit/ 目录 | 未定义 | governance/logs/audit/ | 纳入规范管理 |
| releases/ 目录 | 未定义 | governance/logs/releases/ | 纳入规范管理 |
| lessons/ 目录 | 不存在 | 新增 current/lessons/ | 支持 LESSON-LEARN 机制 |
| 核心文件位置 | 未明确 | 明确仅在根目录 | 遵循 OpenClaw 官方规范 |

## 三、governance 目录详解

### 3.1 总体结构

```
governance/
├── current/                      ← 当前生效版本（静态规范）
│   ├── version_info.yaml
│   │
│   ├── policies/               ← L1：策略和原则层
│   ├── standards/              ← L2：标准和规范层
│   ├── procedures/             ← L3：流程和操作层
│   ├── frameworks/             ← L4：框架和模型层
│   ├── decisions/              ← L5：决策记录层
│   ├── lessons/                ← L6：经验教训层（新增）
│   ├── templates/              ← 模板库
│   ├── best-practices/         ← 最佳实践
│   └── indexes/                ← 索引和导航
│
├── logs/                        ← 动态日志（新增）
│   ├── heartbeat/              ← Heartbeat 日志
│   ├── audit/                  ← 审计日志
│   └── releases/               ← 发布日志
│
└── versions/                    ← 历史版本归档
    └── YYYY.MM.DD_vX.Y/
        └── [同 current 结构]
```

### 3.2 层级定义（L1-L6）

| 层级 | 目录 | 性质 | 变更频率 | 审批要求 |
|:----:|------|------|:--------:|----------|
| **L1** | policies/ | 策略和原则 | 低 | Harold 审批 |
| **L2** | standards/ | 标准规范 | 中 | Harold 审批 |
| **L3** | procedures/ | 流程操作 | 中 | Harold 审批 |
| **L4** | frameworks/ | 框架模型 | 中 | Harold 审批 |
| **L5** | decisions/ | 决策记录 | 高 | 自动记录 |
| **L6** | lessons/ | 经验教训 | 高 | Harold 确认 |

## 四、current/ 目录详解

### 4.1 policies/ - L1 策略和原则层

**用途**：定义掌天智能的组织架构和治理策略

```
policies/
├── README.md
└── ZT-2026-006_组织架构规范.md
```

**⚠️ 重要说明**：
根据 OpenClaw 官方规范，以下文件固定存放在 `${OPENCLAW_LOCAL_WORKSPACE}/` 根目录，**不应在 `.system/governance/` 中重复存储**：
- `SOUL.md` - 核心价值观
- `IDENTITY.md` - Agent 身份定义
- `HEARTBEAT.md` - 银月心跳协议
- `AGENTS.md` - Agent 团队定义
- `BOOTSTRAP.md` - 启动配置
- `TOOLS.md` - 工具使用规范
- `USER.md` - 用户信息

这些文件由 OpenClaw 系统自动加载，是 Agent 运行的核心配置文件。

### 4.2 standards/ - L2 标准和规范层

**用途**：定义数据存储、治理体系等技术标准

```
standards/
├── README.md
├── ZT-2026-001_存储架构规范.md          ← 业务目录存储规范
├── ZT-2026-002_数据治理系统目录规范.md   ← 本文件
└── ZT-2026-004_数据治理体系规范.md       ← 数据分级和隐私规范
```

### 4.3 procedures/ - L3 流程和操作层

**用途**：定义具体的工作流程和 SOP

```
procedures/
├── README.md
├── PROJECT-CREATION-SOP.md              ← Project 创建 SOP
└── ZT-2026-005_工作流程与约束规则.md     ← Task/Topic/Project 管理规范
```

### 4.4 frameworks/ - L4 框架和模型层

**用途**：定义决策框架、优先级框架等可复用模型

```
frameworks/
├── README.md
├── HAROLD-DELEGATION.md                 ← 授权降级框架
├── ZT-2026-011_工作优先级框架.md         ← 多 Project 并行优先级
├── ZT-2026-008_知识增强路径.md           ← 知识→工作增强
└── ADR-001_Agent文件分层架构.md          ← 架构决策记录
```

### 4.5 decisions/ - L5 决策记录层 ⭐

**用途**：存储 HAROLD-DECISION-LIBRARY（DL）

```
decisions/
├── HAROLD-DECISION-LIBRARY-规范.md      ← DL 机制说明
├── HAROLD-DECISION-LIBRARY-索引.md      ← DL 条目索引表（核心）
└── entries/                             ← DL 条目存储
    ├── DL-2026-001-投资决策-早期芯片公司-ACTIVE.md
    ├── DL-2026-002-技术选型-云服务商-ACTIVE.md
    ├── DL-2026-003-人事决策-关键岗位招聘-QUESTIONED.md
    └── ...
```

**设计原则**：
- **信息高度集中**：文件名即内容，包含编号、类型、场景、状态
- **分层加载**：索引表提供快速查找，详细文件提供深入分析
- **表格化索引**：LLM 可快速扫描表格，无需加载所有文件

**文件命名规范**：`DL-YYYY-NNN-{类型}-{场景简述}-{状态}.md`
- 示例：`DL-2026-001-投资决策-早期芯片公司-ACTIVE.md`

**状态定义**：
- `ACTIVE` (ACT) - 活跃可用，正常引用
- `QUESTIONED` (QST) - 被质疑，存在争议
- `REVOKED` (REV) - 已驳回，禁止使用
- `STALE` (STL) - 过时未用，历史保留

**⚠️ 重要**：v1.0 中存在的 `policies/HAROLD-DECISION-LIBRARY.md` 重复文件已在 v2.0 中删除。

### 4.6 lessons/ - L6 经验教训层 ⭐（新增）

**用途**：存储 LESSON-LEARN（LL）条目，支撑 NUCLEUS MODULE 4 的进化机制

```
lessons/
├── LESSON-LEARN-规范.md                 ← LL 机制说明
├── LESSON-LEARN-索引.md                 ← LL 条目索引表（核心）
└── entries/                             ← LL 条目存储
    ├── LL-2026-001-DL误用-Agent执行错误-RESOLVED.md
    ├── LL-2026-002-环境变化-市场周期-ACTIVE.md
    ├── LL-2026-003-环境变化-供应链-RESOLVED.md
    └── ...
```

**设计原则**：
- **信息高度集中**：文件名即内容，包含编号、问题类型、简述、状态
- **分层加载**：索引表提供快速查找，详细文件提供深入分析
- **表格化索引**：LLM 可快速扫描表格，关联 DL 条目

**文件命名规范**：`LL-YYYY-NNN-{问题类型}-{简述}-{状态}.md`
- 示例：`LL-2026-001-DL误用-Agent执行错误-RESOLVED.md`

**状态定义**：
- `ACTIVE` (ACT) - 待处理，需要分析或更新 DL
- `RESOLVED` (RES) - 已解决，DL 已更新或确认无误
- `ARCHIVED` (ARC) - 已归档，历史参考

**问题类型定义**：
- `DL误用` (DL-ERR) - Agent 错误引用或理解 DL
- `环境变化` (ENV) - 外部环境变化导致 DL 失效
- `新场景` (NEW) - DL 未覆盖的新场景
- `例外情况` (EXC) - 合理的例外，需记录

### 4.10 系统级项目目录（projects/）⭐

> **重要**：系统级项目统一存放于 `projects/` 目录下，由 `config/system/system-projects.yaml` 定义。
> 
> **判定标准**（满足任一）：
> - 由 Skill 安装时自动创建
> - 长期运行，无明确结束日期
> - 支撑 OpenClaw 体系的基础运行
> - 由治理版本统一管理
> 
> **命名规范**：
> - yaml 项目 ID：`SYS-*` 前缀（如 SYS-OPS、SYS-GOV）
> - 目录名：小写+连字符（如 sys-ops、sys-gov）
> - 必须有 `dir_path` 字段双保险

**系统级项目清单**（定义于 `config/system/system-projects.yaml`）：

| 项目 ID | 目录名 | Owner | 使命 |
|---------|--------|-------|------|
| SYS-OPS | sys-ops | CTO 菡云芝 | 系统运维（巡检、监控、告警） |
| SYS-GOV | sys-gov | CQO 张铁 | 治理体系维护（PDCA、质量审核） |
| SYS-AGENT | sys-agent | 银月 | Agent 生命周期管理 |
| SYS-DATA | sys-data | CDO 辛如音 | 数据治理（湖仓、备份） |

## 六、文件命名规范

### 6.1 治理文件

**格式**：`ZT-YYYY-NNN_标题.md` 或 `ZT-YYYY-NNN_标题-vX.Y.md`

**示例**：
- `ZT-2026-001_存储架构规范.md`
- `ZT-2026-002_数据治理系统目录规范-v2.0.md`

### 6.2 模板文件

**格式**：`TMPL-{名称}-v{版本}.md`

**示例**：
- `TMPL-PROJECT-CHARTER-v1.2.md`
- `TMPL-周度焦点卡-v1.0.md`（专有名词可保留中文）

### 6.3 决策记录（DL）

**格式**：`DL-YYYY-NNN-{类型}-{场景简述}-{状态}.md`

**示例**：
- `DL-2026-001-投资决策-早期芯片公司-ACTIVE.md`
- `DL-2026-015-技术选型-数据库选择-QUESTIONED.md`
- `DL-2026-007-风险管理-供应链-REVOKED.md`

**类型缩写**：
- INV - 投资决策
- TECH - 技术选型
- HR - 人事决策
- BIZ - 业务决策
- RISK - 风险管理

**状态缩写**：
- ACTIVE (ACT) - 活跃可用
- QUESTIONED (QST) - 被质疑
- REVOKED (REV) - 已驳回
- STALE (STL) - 已过时

### 6.4 经验教训（LL）

**格式**：`LL-YYYY-NNN-{问题类型}-{简述}-{状态}.md`

**示例**：
- `LL-2026-001-DL误用-Agent执行错误-RESOLVED.md`
- `LL-2026-002-环境变化-市场周期-ACTIVE.md`
- `LL-2026-005-新场景-远程招聘-ACTIVE.md`

**问题类型缩写**：
- DL-ERR - DL 误用
- ENV - 环境变化
- NEW - 新场景
- EXC - 例外情况

**状态缩写**：
- ACTIVE (ACT) - 待处理
- RESOLVED (RES) - 已解决
- ARCHIVED (ARC) - 已归档

### 6.5 日志文件

**格式**：`{类型}-LOG-YYYYMMDD.md`

**示例**：

## 七、版本管理

### 7.1 current/ 目录

- 始终保持最新版本
- 所有 Agent 默认引用此目录

### 7.2 versions/ 目录

- 按日期归档历史版本
- 命名格式：`YYYY.MM.DD_vX.Y/`
- 示例：`2026.03.01_v2.8/`

### 7.3 版本变更流程

```
1. 在 current/ 中修改文件
2. 更新 version_info.yaml
3. 归档旧版本到 versions/YYYY.MM.DD_vX.Y/（文件名加版本号）
4. 新版本保持原文件名（不带版本号）
5. 通知相关 Agent
```

> **注意**：GOVERNANCE-MANIFEST.yaml 已于 2026-04-12 废弃，不再需要更新。版本信息由 SKILL.md.metadata.version 和 Git 历史记录。

### 7.4 文件命名与版本号规则 ⭐

**规则**：
- **current/ 目录**：文件名不带版本号
  - 格式：`ZT-YYYY-NNN_标题.md`
  - 示例：`ZT-2026-002_数据治理系统目录规范.md`
  - 版本信息在文件内部 YAML front matter 中体现

- **versions/ 目录**：归档时文件名加版本号
  - 格式：`ZT-YYYY-NNN_标题-vX.Y.md`
  - 示例：`ZT-2026-002_数据治理系统目录规范-v1.0.md`

**原因**：
- 保持引用路径稳定，避免版本更新导致大量引用失效
- current/ 始终指向最新版本
- versions/ 中的版本号便于识别历史版本

**示例**：
```bash
# 当前生效版本（无版本号）
.system/governance/current/standards/ZT-2026-002_数据治理系统目录规范.md

# 历史版本（带版本号）
.system/governance/versions/2026.03.05_v1.0/standards/ZT-2026-002_数据治理系统目录规范-v1.0.md
.system/governance/versions/2026.02.28_v0.9/standards/ZT-2026-002_数据治理系统目录规范-v0.9.md
```
