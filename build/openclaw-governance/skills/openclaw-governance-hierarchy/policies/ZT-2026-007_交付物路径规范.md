# 交付物路径规范 · v1.0

> **文件性质**：治理核心文件 · 交付物存储规范
> **存放路径**：Skill 内置规范文件，运行时由治理框架分发至各 Agent workspace
> **读写权限**：CDO 维护；Harold 审批重大变更
> **上位引用**：ZT-2026-000_SOUL.md · ZT-2026-004_数据治理体系规范.md
> **下位引用**：TMPL-TASK-CARD · HEARTBEAT.md
> **版本**：v1.0
> **变更日期**：2026-03-10

---

## 一、规范目的

确保所有工作交付物存放于规范路径，实现：
1. **知识可沉淀** - 交付物集中在项目目录，知识资产不散落
2. **巡检可验证** - 银月 Heartbeat 可通过路径校验执行情况
3. **审计可追溯** - 每份交付物可追溯至对应 Task-Card

---

## 二、核心原则

### 2.1 Task-Card 强制化

**所有产生交付物的活动（包括创建/更新/删除），必须以 task-card 方式执行。**

| 场景 | 要求 |
|------|------|
| Agent 自主执行任务 | 必须先创建 TASK-CARD |
| 调用子代理 | 必须传递 TASK-CARD（含路径字段） |
| 紧急场景 | 允许"先执行后补卡"，但必须在 48 小时内补充 |

### 2.2 路径唯一性

**交付物必须存放于唯一规范路径，禁止分散存储。**

```
规范路径格式：
10_Projects/{Project-ID}/{Project-Topic}/deliverables/

示例：
10_Projects/ZT-P005_Evolution_of_OpenClaw/ZT-P005-T003_社区最佳实践调研/deliverables/
```

### 2.3 子代理路径传递

调用子代理时，Task-CARD 必须包含以下字段：

| 字段 | 说明 | 必填 |
|------|------|------|
| `deliverable_path` | 完整规范路径 | ✅ |
| `context_refs` | 治理框架加载项 | ✅ |
| `task_type` | 任务类型标识 | ✅ |

## 三、路径结构定义

### 3.1 项目交付物目录

```
10_Projects/
└── {Project-ID}/
    └── {Project-Topic}/
        ├── deliverables/          ← 交付物存放目录
        │   └── *.md / *.pdf
        ├── T001_TASK-CARD.md    ← 任务卡片
        ├── T002_TASK-CARD.md
        └── ...
```

### 3.2 Topic 目录命名规范

> **新增日期**：2026-03-30
> **来源**：SYS-GOV-TP05 规范与 Skill 对齐

**命名格式**：
```
T{两位数字}-{描述}
```

**规则说明**：
| 规则 | 说明 | 示例 |
|------|------|------|
| 序号格式 | T + 两位数字（01-99） | T01, T02, ... T99 |
| 分隔符 | 使用连字符 `-` | T01-Architecture-Design |
| 描述格式 | 英文，首字母大写，连字符连接单词 | T01-Market-Intel |

**正确示例**：
```
10_Projects/ZT-P009_NUCLEUS-2-0/
├── T01-Architecture-Design/
│   ├── TOPIC-BRIEF.md
│   ├── tasks/
│   └── deliverables/
├── T02-Dispatch-Core/
└── T16-Config-Runtime-Loading/
```

**错误示例**：
| 错误格式 | 问题 |
|----------|------|
| `ZT-P000-TP01` | 包含项目前缀，应为 `T01-*` |
| `T01_Market_Intel` | 使用下划线，应为 `T01-Market-Intel` |
| `topic_001_daily_monitoring` | 使用 topic_ 前缀，应为 `T01-Daily-Monitoring` |
| `TP01_内容策划` | 使用 TP 前缀和下划线，应为 `T01-Content-Planning` |
| `topics/T01-*` | 多余的 topics/ 子目录，应直接在项目根目录 |

### 3.3 Agent 工作区（临时）

```
60_Agents/{Agent-ID}/10_Workspace/
├── working/                      ← 仅限临时草稿
│   └── *.md (临时文件)
└── archive/                      ← 已归档历史
```

> **注意**：Agent 工作区仅用于临时草稿，任务完成后必须移至 `deliverables/` 目录。

## 四、Task-CARD 模板要求

### 4.1 必填字段

在 `TMPL-TASK-CARD-任务卡片模板.md` 中，以下字段为必填：

| 字段 | 说明 | 示例 |
|------|------|------|
| `deliverable_path` | 交付物完整路径 | `10_Projects/ZT-P005_.../deliverables/RPT-xxx.md` |
| `deliverable_name` | 交付物名称 | `RPT-技能集成评估报告-v1.0.md` |
| `context_refs` | 治理框架引用 | `ZT-2026-001~008` |

### 4.2 路径校验逻辑

```
Task-CARD 创建时：
  ↓
检查 deliverable_path 是否符合规范格式？
  ↓
  ├─ 是 → 允许创建 Task-Card
  └─ 否 → 拒绝创建，提示修正
```

## 五、银月巡检验证（HEARTBEAT Step 3）

### 5.1 路径校验检查项

银月在 Heartbeat Step 3 验收时，增加以下检查项：

| 检查项 | 动作 |
|--------|------|
| `deliverable_path` 字段存在 | Task-Card 必须包含路径字段 |
| 路径符合规范格式 | `10_Projects/{Project-ID}/*/deliverables/` |
| 交付物实际存在于路径 | 核对文件系统 |
| 无违规存放在 Agent 工作区 | 扫描 `60_Agents/*/10_Workspace/working/` |

### 5.2 异常处理

| 异常类型 | 处理方式 |
|----------|----------|
| 无 Task-Card 但有 deliverable | 标记 `[!]` 待修订，要求补充 Task-Card |
| deliverable 在错误路径 | 标记 `[*]` 阻塞，移至正确路径后消除 |
| 路径格式不符合规范 | 标记 `[!]` 待修订，通知 Task PM 修正 |
