## 一、工作结构层次

掌天智能的工作遵循严格的四级层次结构。**高层级包含低层级，低层级不得脱离高层级独立存在。**

```
Project（项目）
 └── Topic（主题）
      └── Task（任务）
           ├── Sub-Task（子任务）
           └── Deliverables（交付物）
```

### 1.1 Project（项目）

| 属性               | 说明                                      |
| ------------------ | ----------------------------------------- |
| **定义**     | 顶层工作集，聚焦一个大目标                |
| **协作模式** | 多 Agent 协作                             |
| **PIC**      | 由 Harold 指定                            |
| **进度跟踪** | PMO 银月 统一负责所有 Project 的进度跟踪  |
| **创建权限** | 仅 Harold 可发起                          |
| **关闭权限** | 仅 Harold 可关闭                          |
| **输入**     | Harold 的任务指令、战略目标、相关背景资料 |
| **输出**     | 包含若干 Topic                            |

**MUST**：

- 每个 Project 必须拥有唯一编号（如 ZT-P001）并注册到项目注册表
- Project PIC 必须记录来自 Harold / 银月 的评审意见，并据此进行改进

**MUST NOT**：

- 无 Harold 批准，任何 Agent 不得自行创建 Project

---

### 1.2 Topic（主题）

| 属性               | 说明                               |
| ------------------ | ---------------------------------- |
| **定义**     | Project 的子模块，聚焦一个具体主题 |
| **协作模式** | 多 Agent 协作                      |
| **Main PIC** | 由 Project PIC 或 Harold 指定      |
| **创建权限** | Project PIC 或 Harold              |
| **关闭权限** | **仅 Harold 或 银月** 可关闭 |
| **输入**     | Project 目标的分解 + 引用相关 Task |
| **输出**     | 包含若干 Task                      |

**Topic Main PIC 的职责**：

1. 在 Topic 范围内创建 Task 并指定 Task PIC
2. 管理 Topic 下所有 Task 的进度与质量
3. 对 Topic 的整体交付负责，直到 Topic 被关闭
4. 记录来自 Harold / 银月 的评审意见，并据此进行改进

**MUST**：

- 每个 Topic 必须有唯一编号（如 ZT-P001-TP01）
- Topic Main PIC 不得将关闭权限委托给他人

**MUST NOT**：

- Topic Main PIC 以外的 Agent 不得在该 Topic 下创建 Task（除非获得 Main PIC 显式授权）
- 任何 Agent 不得自行关闭 Topic（即使所有 Task 已完成）

---

### 1.3 Task（任务）

| 属性               | 说明                                                                       |
| ------------------ | -------------------------------------------------------------------------- |
| **定义**     | Topic 的执行单元，聚焦一个具体行动                                         |
| **协作模式** | **单一 PIC**，可拆分为若干 Sub-Task                                  |
| **PIC**      | 由 Topic Main PIC 或 银月 指定                                             |
| **创建权限** | Topic Main PIC 或 银月                                                     |
| **关闭权限** | **Topic Main PIC 或 银月**                                           |
| **输入**     | 上游 Task 的输出、参考资料、模板（**输入是工作流的最小可引用单元**） |
| **输出**     | Deliverables                                                               |

**Task PIC 的职责**：

1. 执行 Task 及其 Sub-Task
2. 确保 Deliverables 质量符合标准
3. 记录来自 Harold / 银月 的评审意见，并据此进行改进
4. Task 完成后，向 Topic Main PIC 提交关闭申请

**MUST**：

- 每个 Task 必须有唯一编号（如 ZT-P001-TP01-T001）
- 每个 Task 必须有且只有一个 PIC
- Task 的输入必须通过 @ref 引用可追溯的来源（最小单元是另一个 Task 或其 Deliverable）
- **所有未完成的 Task 必须计入该 PIC 的 `MISSION_BOARD.md`**

**MUST NOT**：

- 非 PIC 不得修改该 Task 的 Deliverables（除非发生正式任务移交）
- PIC 不得自行关闭 Task，必须由 Topic Main PIC 或 银月 关闭

---

### 1.4 Sub-Task（子任务）

| 属性               | 说明               |
| ------------------ | ------------------ |
| **定义**     | Task 的细分步骤    |
| **PIC**      | 继承父 Task 的 PIC |
| **创建权限** | Task PIC           |
| **关闭权限** | Task PIC           |

**MUST**：

- Sub-Task 编号继承父 Task（如 ZT-P001-TP01-T001-S01）

**MUST NOT**：

- Sub-Task 不得独立于 Task 存在

## 二、输入物管理

> **核心原则**：无输入，不执行。每一份输出都必须能追溯到输入来源。

### 2.1 输入引用规则

| 规则                       | 说明                                                                    |
| -------------------------- | ----------------------------------------------------------------------- |
| **最小引用单元**     | Task（或 Task 的 Deliverable）                                          |
| **引用方式**         | 通过 @ref 标注来源 Task 编号或 Deliverable 编号                         |
| **可接受的输入类型** | 上游 Task 的 Deliverable、Harold 的指令、外部数据（需标注来源与置信度） |

**MUST**：

- 每个 Task 启动前，PIC 必须确认输入物已就绪且状态为 Active
- 输入物变更时，下游 Task PIC 必须收到通知并评估影响

**MUST NOT**：

- 引用状态为 Draft / Stale / Archived / Destroyed 的文件作为正式输入
- 使用无法追溯来源的数据作为输入（零容忍）

## 三、交付物管理（Deliverables）

### 3.1 生命周期规则（REF-001）

所有 Deliverables 必须严格遵循以下状态流转：

| 流转                         | 触发条件                                                      | 执行人           | 审批人            | 禁止行为             |
| ---------------------------- | ------------------------------------------------------------- | ---------------- | ----------------- | -------------------- |
| **Draft → Active**    | 元数据齐全 + Quality Gate 通过 + 评审记录（decision=approve） | PIC              | CDO               | 无评审直接激活       |
| **Active → Stale**    | 90 天未更新，且无有效豁免记录                                 | 系统扫描（每周） | CDO 确认          | Agent 手动标记 Stale |
| **Stale → Archived**  | 进入 Stale 超 30 天未处理                                     | CTO 执行         | CDO 批准          | 无批准归档           |
| **Archived → Active** | 内容更新 + 重新通过 Quality Gate                              | PIC              | CDO               | 跳过评审直接恢复     |
| **Any → Destroyed**   | 保留期满 + 二人复核 + 审计记录（保留 3 年）                   | CTO 执行         | Harold + CDO 双签 | 单人销毁             |

**MUST**：

- 状态变更前，必须验证当前状态是否允许目标流转（禁止跳跃）
- 每次状态变更必须生成变更记录（who / when / from_status / to_status / reason）

**MUST NOT**：

- 跳过中间状态（如 Draft → Archived）
- 引用 Stale / Archived / Destroyed 状态的 Deliverable 作为决策依据

### 3.2 文档命名规范

所有 Deliverables 必须遵循统一命名规则：

```
{项目编号}-{Topic编号}-{Task编号}-{交付物类型}-{标题简称}-v{版本号}

示例：
ZT-P001-TP01-T001-RPT-供应商A尽调报告-v1.0
ZT-P001-TP02-T003-TMPL-周报模板-v2.1
```

**命名要素**：

| 要素       | 格式       | 说明                                                                 |
| ---------- | ---------- | -------------------------------------------------------------------- |
| 项目编号   | ZT-P{3位}  | 如 ZT-P001                                                           |
| Topic 编号 | TP{2位}    | 如 TP01                                                              |
| Task 编号  | T{3位}     | 如 T001                                                              |
| 交付物类型 | 受控代码   | RPT=报告, TMPL=模板, SPEC=规范, DATA=数据, MEMO=备忘录, MIN=会议纪要 |
| 标题简称   | 中文/英文  | 简洁描述内容                                                         |
| 版本号     | v{主}.{次} | 主版本=重大修订, 次版本=小修                                         |

**MUST**：

- 文件名中不得包含空格（使用连字符 `-` 分隔）
- 版本号必须单调递增

**MUST NOT**：

- 使用"最终版""final""new"等非结构化版本标记

### 3.3 文档格式规范

每份 Deliverable 文件必须包含以下标准文档头：

```markdown
---
doc_id: ZT-P001-TP01-T001-RPT-供应商A尽调报告-v1.0
status: draft
privacy_level: P2
pic: CDO_xinruyin
project: ZT-P001
topic: ZT-P001-TP01
task: ZT-P001-TP01-T001
created_at: 2026-02-21
updated_at: 2026-02-21
feishu_link: https://xxx.feishu.cn/docx/xxxxx
review_records:
  - reviewer: PA_yinyue
    date: 2026-02-22
    decision: approve / reject / revise
    comments: "评审意见内容"
inputs:
  - ref: ZT-P001-TP01-T002-DATA-供应商A财务数据-v1.0
---
```

## 四、模板管理

> **核心原则**：无模板，不开工。模板是交付物质量的底线保障。

### 4.1 模板治理规则

| 规则编号 | 规则内容                                                                                           |
| -------- | -------------------------------------------------------------------------------------------------- |
| TMPL-001 | 所有交付物模板**由 CDO 领导的数据治理委员会统一创建和管理**                                  |
| TMPL-002 | **任何 Agent 无权私自创建模板**                                                              |
| TMPL-003 | 新模板需求**由使用方提出申请**，经数据治理委员会评审通过后方可发布                           |
| TMPL-004 | 若某 Task 所需的模板尚未发布，**该 Task 状态必须标记为 BLOCKED**，PIC 不得使用非标准格式替代 |
| TMPL-005 | 已发布模板的修订，需提交数据治理委员会重新评审                                                     |

### 4.2 模板申请流程

```
使用方（任意 Agent）发现需要新模板
        │
        ▼
提交模板需求申请（描述用途、字段需求、使用场景）
        │
        ▼
CDO 辛如音 受理，提交数据治理委员会评审
        │
        ▼
  ┌─────────────────────────────────────┐
  │ 评审通过 → CDO 创建模板 → 发布      │
  │ 评审不通过 → 退回并说明理由          │
  │ 需修改 → 与使用方沟通后重新提交      │
  └─────────────────────────────────────┘
        │
        ▼
模板发布后，BLOCKED 的 Task 解除阻塞
```

### 4.3 模板版本管理

- 模板文件命名：`TMPL-{类型代码}-{名称}-v{版本号}`
- 旧版模板不得删除，标记为 `deprecated` 并归档
- 已使用旧版模板创建的 Deliverables 无需追溯更新，但新创建的 Deliverables 必须使用最新版本

### 5.1 飞书文档规则

## 六、评审与反馈闭环

> **核心原则**：评审意见不是"已阅"，而是"必须响应"。

### 6.1 评审规则

| 规则编号 | 规则内容                                                                     |
| -------- | ---------------------------------------------------------------------------- |
| REV-001  | Harold 和 银月 有权对任何层级（Project / Topic / Task）的工作发起评审        |
| REV-002  | PIC 必须在收到评审意见后**记录**评审内容（reviewer / date / comments） |
| REV-003  | PIC 必须针对评审意见**制定改进措施**并执行                             |
| REV-004  | 改进完成后，PIC 必须向评审人**回复改进结果**                           |
| REV-005  | 未响应的评审意见将被标记为 OVERDUE，计入 Agent 待办（MISSION_BOARD.md）      |

### 6.2 评审记录格式

每条评审意见必须记录以下信息：

| 字段          | 说明                                    |
| ------------- | --------------------------------------- |
| reviewer      | 评审人（Harold / 银月）                 |
| date          | 评审日期                                |
| target        | 评审对象（Project / Topic / Task 编号） |
| decision      | approve / reject / revise               |
| comments      | 具体意见                                |
| response      | PIC 的响应（改进措施 + 完成状态）       |
| response_date | 响应日期                                |

## 七、待办跟踪（MISSION_BOARD.md）

> **核心原则**：未完成的事项不会自动消失，必须被看见。

### 7.1 MISSION_BOARD 规则

| 规则编号 | 规则内容                                                                                            |
| -------- | --------------------------------------------------------------------------------------------------- |
| MB-001   | 每个 Agent 必须维护自己的 `MISSION_BOARD.md`                                                      |
| MB-002   | **所有未完成的 Task（含 Sub-Task）必须出现在 PIC 的 MISSION_BOARD.md 中**                     |
| MB-003   | **所有未响应的评审意见必须出现在 PIC 的 MISSION_BOARD.md 中**                                 |
| MB-004   | **所有因无模板而 BLOCKED 的 Task 必须出现在 PIC 的 MISSION_BOARD.md 中**，并标注 BLOCKED 原因 |
| MB-005   | Task 关闭后，从 MISSION_BOARD.md 中移除                                                             |
| MB-006   | PMO 银月 有权随时检查任何 Agent 的 MISSION_BOARD.md                                                 |

### 7.2 MISSION_BOARD 条目格式


## 九、异常处理

| 异常场景                      | 处理规则                                                |
| ----------------------------- | ------------------------------------------------------- |
| Task 超期无进展（>7天）       | PIC 必须主动上报 Topic Main PIC；银月 有权介入          |
| 输入物状态变为 Stale          | 下游 Task PIC 必须暂停并评估影响，上报 Topic Main PIC   |
| 两个 Agent 对同一数据结论冲突 | 禁止自行覆盖，上报 CDO 仲裁                             |
| 模板缺失导致 BLOCKED          | PIC 立即提交模板申请，并在 MISSION_BOARD 中标注 BLOCKED |
| PIC 不可用（如 Agent 离线）   | 银月 有权临时指定代理 PIC，并记录移交日志               |

## 十、状态传播规则（新增）

> **核心原则**：状态变更自动同步，派生视图自动更新。"文件是唯一真相"的落地机制。

### 10.1 状态传播链路

```
┌─────────────────────────────────────────────────────────────────┐
│  Task-CARD.md (数据源)                                          │
│  status: draft → active → pending_review → done               │
└───────────────────────────────┬─────────────────────────────────┘
                                │ 自动同步
                                ↓
┌─────────────────────────────────────────────────────────────────┐
│  TOPIC-BRIEF.md (派生视图)                                       │
│  Task 列表自动汇总状态：X active, Y done, Z blocked             │
└───────────────────────────────┬─────────────────────────────────┘
                                │ 自动同步
                                ↓
┌─────────────────────────────────────────────────────────────────┐
│  PROJECT-CHARTER.md (派生视图)                                  │
│  Topic 列表自动汇总状态：X active, Y done                       │
└───────────────────────────────┬─────────────────────────────────┘
                                │ 自动同步
                                ↓
┌─────────────────────────────────────────────────────────────────┐
│  MISSION_BOARD.md - 银月 Overview                               │
│  所有 Project 状态汇总，阻塞告警                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 10.2 Agent MISSION_BOARD 跟踪范围

每个 Agent（c[x]o）的 MISSION_BOARD 必须跟踪以下 4 个层级：

| 层级 | 跟踪内容 | 状态字段 |
|------|---------|---------|
| **Project** | 参与的 Project 整体状态 | status, progress |
| **Topic** | 参与负责的 Topic 进度 | task_count, done_count |
| **Task** | 自己作为 PIC 的 Task | status, due_date |
| **Deliverable** | 待审批/待验收的交付物 | status, review_status |

### 10.3 同步规则

| 事件 | 触发动作 | 执行人 | 时限 |
|------|---------|--------|------|
| Task 状态变更 | TOPIC-BRIEF 自动汇总 | 系统/扫描脚本 | 2 小时内 |
| Topic 内 Task 状态变更 | PROJECT-CHARTER 自动汇总 | 系统/扫描脚本 | 2 小时内 |
| Topic 状态变更 | MISSION_BOARD 自动汇总 | 银月扫描脚本 | 每次 Heartbeat |
| 阻塞状态出现 | 向上级 PIC 告警 | 系统 | 立即 |

### 10.4 禁止事项
